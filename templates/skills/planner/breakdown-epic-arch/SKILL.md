# Breakdown Epic Architecture — アーキテクチャ分解フレームワーク

大規模コマンド（複数レイヤー・複数コンポーネントにまたがる変更）を、依存関係に基づいて安全に分解する。

## システムアーキテクチャ分析

分解前に以下を特定する:

| 分析項目 | 内容 |
|---------|------|
| **影響レイヤー** | データ層 / ドメイン層 / インターフェース層 / インフラ層 |
| **依存方向** | 下位→上位（通常）/ 共有→利用側（横断的） |
| **変更の種類** | 新規追加 / 既存修正 / リファクタリング |

## Phases 構造

### Concrete Phase（確定フェーズ）
即座に実行可能なタスク群。調査結果に依存しない。

- 基盤となるデータモデル・型定義
- 既存コードの前提条件が明確な変更
- 設定ファイルの追加・変更

### Deferred Phase（後続フェーズ）
前フェーズの結果に基づいて詳細化するタスク群。

- 調査結果に依存する実装
- 統合テスト・E2Eテスト
- リファクタリング（影響範囲確定後）

## Wave 構造との連携

各 Phase 内で並列実行可能なタスクを Wave にグループ化する:

```
Phase 1 (concrete)
  Wave 1: [調査A, 調査B]          # researcher × 並列
  Wave 2: [設計]                   # architect
  Wave 3: [実装A, 実装B, 実装C]   # implementer × 並列
  Wave 4: [検証]                   # quality-assurance

Phase 2 (deferred) — Phase 1 完了後に詳細化
  Wave 5: [統合実装]
  Wave 6: [統合テスト]
```

## 段階的委譲パターン

大規模変更では以下の順序で persona を割り当てる:

| 順序 | persona_hint | 作業内容 |
|------|-------------|---------|
| 1 | researcher | 既存構造把握、影響範囲特定、技術的制約の洗い出し |
| 2 | architect | 設計判断、インターフェース定義、データモデル策定 |
| 3 | implementer | コード実装、ユニットテスト作成、ドキュメント更新 |
| 4 | quality-assurance | 統合テスト、コードレビュー、セキュリティ確認 |

## レイヤー間の分解原則

- **下位レイヤーを先に**: データモデル → ドメインロジック → API の順
- **インターフェースは最後**: 内部が安定してから外部公開
- **共有コンポーネントは独立タスク**: 複数箇所から参照される変更は単独タスク化
- **マイグレーションは分離**: スキーマ変更は専用タスクとして切り出す
