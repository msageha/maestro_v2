# Security Threat Model Framework

Planner がセキュリティ分析タスクを Maestro タスクに分解するための設計フレームワーク。

## 1. 8ステップワークフロー → タスク分解パターン

### Phase 1: 調査（researcher ペルソナ）

| Step | 内容 | タスク設計指針 |
|------|------|--------------|
| **Step 1: スコープ抽出** | システムモデルの特定（コンポーネント、データフロー、外部連携） | **機械的リポジトリ実体列挙（→ §1.1）を最初の必須手順とする**。その上で `researcher` ペルソナの網羅走査タスクを設計し、content に対象ディレクトリ・技術スタックを明記 |
| **Step 2: 境界・資産・エントリポイント導出** | 信頼境界、保護対象資産、攻撃面の列挙 | Step 1 の結果を依存タスクとして設計。acceptance_criteria に資産カテゴリ網羅（§2）を要求 |
| **Step 3: 資産の重要度と攻撃者能力の評価** | 資産価値と想定攻撃者のキャリブレーション | Step 2 依存。リスク優先度表（§3）を criteria に含める |

### §1.1 Step 1 の必須手順: 機械的リポジトリ実体列挙

スコープ抽出は **人手の俯瞰ではなく、機械的なファイル実体の全列挙を起点とする**。LLM worker は自然言語の対象記述だけで走査範囲を決めると、インフラ/設定ファイル群を「リポジトリ外」と誤って公理化し、その前提が後続 worker と最終レポートに伝播する（集約層はこの未走査領域を事後復元できない）。これを構造的に断つため、Step 1 のタスク `content` / `acceptance_criteria` に以下を必須化する:

1. **トラッキング対象の全列挙**: `git ls-files` でリポジトリにコミットされた全ファイルを列挙し、成果物に添付する。ディレクトリツリー（`git ls-files | sed 's#/[^/]*$##' | sort -u` 等）も取得して構造を把握する。
2. **インフラ/CI/設定ファイルの網羅検出**: 少なくとも以下のグロブを明示的に走査し、ヒットしたパスを一覧化する — `*.tf` / `*.tfvars`（Terraform）, `*.yaml` / `*.yml`（CloudFormation / k8s / CI / config）, `Dockerfile*` / `docker-compose*`, `.github/workflows/*`, `.gitlab-ci.yml`, `.npmrc`, `.env*`, `*.toml`（設定）。
3. **「リポジトリ外」分類には否定の根拠を必須化**: ある資産・防御層を「範囲外」「リポジトリに存在しない」と分類する場合、**機械的列挙の結果に当該パスが存在しないこと（grep / ls-files の空結果）を否定の根拠として成果物に明記する**。否定の根拠なしに範囲外宣言してはならない（これが「インフラ層まるごと欠落」の直接原因である）。

acceptance_criteria 例:
```
- `git ls-files` の全件出力が成果物に含まれている
- `*.tf` / `.github/workflows/*` / `Dockerfile*` / `.env*` 等のインフラ・CI・設定ファイルが走査され、ヒットした全パスが一覧化されている
- 「リポジトリ外」と分類した資産には、機械的列挙に該当パスが存在しないことの否定の根拠（空の grep/ls-files 結果）が添えられている
```

### Phase 1 並行: 横断 sweep（sweeper ペルソナ）

domain charter（認証・入力・ネットワーク等）に分解した worker は、専門特化の深さと引き換えに「どの領域にも明確に属さない seam 案件」を構造的に取りこぼす（§3 で Low が任意扱いになることと、researcher / QA の stay-in-lane 規則が重なり、領域横断の偶発的発見が抑制される）。これを補うため、**全 domain worker と並行する単一の横断 sweep タスクを 1 つ追加する**:

- **ペルソナ**: `sweeper`。リポジトリ全体を 1 つのメンタルモデルで通読し、seam / 低深刻度案件を拾い上げる。
- **`required: true` を必須**: §3 で Low が任意扱いでも、この sweep タスクだけは required にし「低深刻度でも seam 案件は必ず一度は通読する」を担保する。
- **観点**（content に明記）: コミット済みの不要・危険ファイル（`*.log` / ローカルパス混入）、未検証の入力パラメータ、休眠中だが危険な残存ヘルパ、設定の foot-gun、ファイル衛生。
- **重複許容**: 他 domain worker と重複する所見が出てもよい（dedup 機構が吸収する）。落とすより拾うを優先する。
- **expected_paths**: リポジトリ全体を通読するため `["."]` を指定する。

タスク設計例:
```yaml
- name: "cross-cutting-sweep"
  purpose: "domain charter に属さない seam / 低深刻度の所見を横断的に拾い上げる"
  content: "リポジトリ全体を 1 つのメンタルモデルで通読し、seam 案件を検出する。観点: コミット済みの不要・危険ファイル（*.log / ローカル絶対パス混入）、バリデーションなしに使われる入力パラメータ、休眠中だが危険な残存ヘルパ、デフォルトで危険な設定、ファイル衛生。専門深掘りは他 worker に委ね、seam の存在に気付きレポートに載せるところまでを担保する。各所見に file:line 根拠を添える"
  acceptance_criteria: "低深刻度を含む seam 案件が file:line 根拠付きで列挙されている。各所見に『どの domain charter にも属さない/複数領域に跨る理由』が一言添えられている"
  bloom_level: 4
  persona_hint: "sweeper"
  skill_refs: ["source-grounded-response"]
  required: true
  expected_paths: ["."]
  definition_of_abort:
    max_repair_count: 3
    max_wall_clock_sec: 1800
```

### Phase 2: 分析（researcher ペルソナ）

| Step | 内容 | タスク設計指針 |
|------|------|--------------|
| **Step 4: 脅威列挙** | 悪用パスの網羅的列挙（STRIDE/攻撃ツリー） | 各脅威にリポジトリパス参照を要求（エビデンスベース） |
| **Step 5: 優先度付け** | 発生可能性×影響度の推論による優先順位決定 | §3 のリスク優先度に基づき required フラグを判定 |

### §1.4 各 domain worker の深度・接地の常設責務（depth-by-default）

集約のコード接地（§1.3）を統合 worker だけに任せず、**各 domain worker にも自分の finding を深く・コードに接地させる**ことを常設責務にする:

- **採用 finding の自己裏取り**: High 以上の finding は worker summary に推測で書かず、**worker 自身が当該 file:line を開いて確認**してから載せる。確認結果と異なる場合は載せない。
- **自タスク内の深掘りは推奨される積極行動**: researcher / quality-assurance の stay-in-lane 規則（タスク範囲外は停止）は **タスク範囲外への越境を止めるもので、自タスク内の深掘りを抑制するものではない**。担当 domain 内では「もう一段深く確認する」を既定の積極行動とし、浅い列挙で満足（satisfice）しない。
- **実証は環境が許せば**: 動的に検証可能な High 以上の脅威は、実行環境が許せば probe で実証ステータス（実証済 / コード確証 / 理論段階）を付ける。probe の実走ロケーションは §1.5。実証できない環境では理論段階のステータスを正直に付け、§3.1 の重大度上限ガードに従う（未実証を実証と偽らない）。

### §1.5 hot-finding の機会的深掘り（再 plan ループ）と probe の実走ロケーション

worker が事前計画になかった「深掘り価値の高い finding」を見つけたときに深掘りを注入する経路を制度化する。single-agent の「気づいた瞬間に深掘りする」を、Daemon 往復ありの再 plan として近似するもの:

- **worker 側（推奨の構造化報告）**: worker は `maestro plan add-task` を直接呼べない（Planner 所有）。深掘り価値の高い finding を見つけたら、`maestro result write` の summary に **「深掘り推奨（対象 file:line / 想定 probe・アプローチ / 期待される確証）」を構造化して報告**する。
- **Planner 側（注入判断）**: Planner は結果を読み、価値が高ければ `maestro plan add-task --command-id <id> --operation-type verify ...` で深掘り / 実証タスクを注入する（`operation_type: verify` を明示しないと既定が repair 扱いになり FAIL 時に無意味な自動リトライが走る）。既存の add-task 上限（`max_tasks` 超過は reject）を尊重し、同一 finding への深掘りを無限に積まない。
- **probe の実走ロケーション**: probe 実走が必要な場合は、最終 phase に **`run_on_integration: true` + `operation_type: verify` の検証タスク**として配置できる。worker worktree は gitignored な依存キャッシュを持たないため、private 依存を要する probe はこの場で走らせる（operator が integration worktree に deps を用意する前提。詳細は planner instructions の verification タスク節）。**ただし §1.3 の永続化制約に注意**: `run_on_integration` で書いた probe スクリプト・観測ファイルは pre-merge auto-clean で削除される。probe スクリプト本体と観測結果（fired 件数 / audit 全件カウントの critical・high 内訳）は、**通常 worker worktree タスクで生成・コミットした成果物（findings / probes ディレクトリ）に必ず残す**。run_on_integration タスクは「deps のある環境で実行して観測する」役割に限定し、観測値は result summary 経由で committed 成果物に反映する。

### Phase 3: 検証（quality-assurance ペルソナ）

| Step | 内容 | タスク設計指針 |
|------|------|--------------|
| **Step 6: 前提検証** | ユーザーとの仮定の妥当性確認 | 分析結果のレビュータスクとして設計 |
| **Step 7: 緩和策の推奨** | 対策と優先対応パスの提示 | セキュリティコントロールカテゴリ（§5）を参照 |
| **Step 8: 品質チェック + 最終統合** | 脅威モデル全体の整合性・網羅性検証と最終散文レポートの生成 | 最終検証タスク。全 Step の成果物を突合し、加えて §1.2 の **事実性の独立再導出 + 未走査領域の差分照合**、§1.3 の **code-read による最終統合（採用 finding の file:line 再確認 / cross-report 突合 / 重大度の最終確定）**を必須化 |

### §1.2 Step 8 の拡張: 事実性 + 未走査領域の独立再検証

成果物どうしの突合（プロセス整合）だけでは、(1) 形式的に基準を満たしたが事実として誤った主張、(2) そもそも誰も走査していない領域、を検出できない。Step 8 の品質チェックタスク（`quality-assurance` ペルソナ）に既存の Turn-2 検証（推測ベース主張の検出）を維持しつつ以下 2 観点を**追加**する:

1. **主張の独立再導出（事実性）**: 「修正済み / 該当なし / N/A / 影響なし」と結論した主要主張を、QA が**別経路で独立に再検証**する。依存については「個別パッケージ + 単一 CVE 確認」だけで「修正済み」と結論することを禁じる。検証手段は実行環境の能力に応じて段階的に選ぶ:
   - **監査ツールが完走できる環境**: `npm audit` / `pip-audit` / `osv-scanner` 等で**全件カウント**（critical / high の内訳を含む）を取得し成果物に含める。
   - **完走できない環境**（worker worktree は gitignored な依存キャッシュを持たず、無人運用で Bash 監査ツールを回せないことがある）: 取得不能であることを**明示した上で satisfice しない**。単一 CVE での「修正済み」結論には進まず、「全件監査が未実施のため事実性は未確証」と制約を記し、当該主張を **`要追加検証` 扱い（必要なら `--status failed`）で返す**。

   いずれの場合も「同バージョンに別 high 勧告がオープンなのに単独 CVE で修正済みと誤判定」する型の事故を、**未確証を確証と偽らない**ことで遮断する。完走できない環境で全件監査を捏造・推測で埋めてはならない。
2. **未走査領域の検出（網羅性）**: §1.1 の機械的ファイル列挙（`git ls-files` の全件）と、各 worker が実際に参照した file:line の集合を**差分照合**し、走査されていないディレクトリ / ファイル群を「未カバー領域」として表で明示報告する。未カバーが残る場合は **`--status failed`** で返し、未カバーのパス群を summary に列挙して Planner に再分解を促す。

acceptance_criteria 例:
```
- 依存検証は全件カウント（npm audit 等の総数 + critical/high 内訳）を成果物に含む。取得不能な環境では未実施を明示し、単独 CVE での「修正済み」結論に代えて `要追加検証` で返している
- 「走査済みファイル集合 vs git ls-files 全件」の差分が表として出力され、未カバー領域（例: terraform/）が検出されていれば --status failed で返している
```

### §1.3 最終統合レポートは code-read worker が再導出する

集約層（Orchestrator / Planner）はコードを読めず（README「ロール別の権限」: コードの読み書き不可）、Agent 間ハンドオフはすべて Daemon 経由のテキスト成果物に非可逆シリアライズされる。このため最終レポートを上位層の暗黙の要約に任せると、worker summary の転記に留まり、コードに戻って主張を裏取り・自己訂正できない。これを補うため、**最終散文レポートの生成を Step 8 と同 phase の code-read worker タスクとして明示発注**する（`quality-assurance` ペルソナの Step 8 を執筆まで拡張するのが既定。専用ペルソナは増やさない）。

> **重要（成果物の永続化制約 — これを誤ると最終レポートが消える）**: 最終レポートやその他の**成果物ファイルは必ず通常の worker worktree タスクで生成**する（daemon が worktree を `git add -A` で auto-commit → integration へ merge → publish するため永続化される）。**`run_on_integration: true` タスクで成果物ファイルを書いてはならない** — integration worktree の未コミット変更は publish 直前の `integration_pre_merge_autoclean` が「検証の一時生成物（build 出力等）」とみなして**削除**するため、そこに書いたレポートは publish されず失われる。`run_on_integration` は **lint/check/probe 等の検証コマンドの実行（観測結果は通常 worker タスク側の成果物に記録）専用**であり、レポート本体・findings 等の deliverable の生成場所ではない。最終レポートを生成する synthesis タスクは `run_on_integration: false`（通常 worker worktree）で発注し、その worktree にレポートファイルを書く。

§1.2 の 2 観点に加えて以下を必須化する:

3. **採用 finding の独立再確認（コード接地）**: High 以上の主要 finding は worker summary をそのまま転記せず、**統合 worker 自身が当該 file:line を開いて再確認**してから載せる。上流主張と差異があれば「訂正」セクションに理由付きで明記する（上流 worker の satisfice した誤判定 — 単独 CVE での「修正済み」型 — を最終レポートに素通りさせない）。
4. **cross-report 突合（既存 / 別手法レポートが入力にある場合のみ）**: 既存の評価レポートや別手法の出力が提供されたときは、finding 単位で (i) 一致 / (ii) 片側のみ検出 / (iii) 矛盾 に分類する。(ii)(iii) は「降格 / 訂正 / 追加調査」候補として、コードで裏取りした結論を理由付きで記載する。「相手にあって自分に無い所見」は**自分の取りこぼし検出**にも使う。入力が無い場合はこの観点はスキップしてよい。
5. **重大度の最終確定（grounding owner）**: 各 domain worker が付けた暫定重大度を、統合 worker が **§3.1 の検証ステータス上限ガードと併せて最終確定**する。domain をまたぐ連鎖の重大度を、実証有無を見ずに理論最悪値で集約しない。

acceptance_criteria 例:
```
- 最終レポートの High 以上の所見に、統合タスクが独立に開いて確認した file:line 参照が伴い、上流との差異は「訂正」に明記されている
- 既存レポートを入力に与えた場合のみ、突合表（一致 / 片側 / 矛盾）が出力され、矛盾項目にコード根拠付きの結論が付いている
- 連鎖 finding の重大度が §3.1 と整合し、その連鎖の最低検証ステータスを超えていない
```

## 2. 資産カテゴリ

タスクの acceptance_criteria で網羅性を確認する際のチェックリスト:

- **ユーザーデータ**: PII、認証情報、セッション
- **認証アーティファクト**: JWT、OAuth トークン、API キー
- **シークレット/鍵**: 暗号鍵、サービスアカウント、証明書
- **設定**: 環境変数、設定ファイル、フィーチャーフラグ
- **ソースコード**: ビジネスロジック、アルゴリズム、IP
- **インフラ as code（IaC）/ エッジ防御 / CI 設定**: Terraform / CloudFormation 等の IaC、WAF / Cloud Armor / body 検査ポリシー、LB / ingress、ストレージバケットポリシー、`.github/workflows/` 等の CI 設定、Dockerfile / コンテナ定義、`.npmrc` 等のレジストリ設定（§1.1 の機械的列挙でヒットしたファイルを必ずこのカテゴリに分類する）

## 3. リスク優先度 → required フラグ判定

| 優先度 | 脅威例 | タスク設計 |
|--------|--------|----------|
| **High** | RCE、認証バイパス、権限昇格、SQLi | `required: true` — 必ず分析・対策タスクを生成 |
| **Medium** | DoS、部分的情報露出、CSRF、セッション固定 | 状況に応じて required 判定 |
| **Low** | 情報リーク（バージョン露出等）、冗長ログ | 任意タスクとして追加 |

### §3.1 検証ステータスによる重大度上限ガード

重大度は「理論上の最悪値」ではなく「**どこまで成立を確認したか**」で上限を切る。動的に成立を実証していない攻撃連鎖（chain）に最高ランクを付与すると、未実証の前提を積み上げた理論最悪値で重大度が上振れする（本文では「最終成立はモデル挙動依存で未実証」と告白しつつ Critical を付ける、という型の事故）。

- **Critical を付与できるのは次のいずれかのみ**: (1) 動的検証で成立を**実証済み**、または (2) 実装単体で前提条件なしに成立することが**コード確証できる**もの。
- **検証ステータスが「コード確証どまり」または「理論段階」に留まる連鎖は、最高でも High + `要追加検証` ラベル**とする。本文に「最終成立はモデル挙動依存で未実証」「前提条件次第」等と書かれる所見を Critical で出してはならない。
- multi-agent の検証観は「コード正しさ（build/lint/test）」寄りで PoC 実走を伴わないことが多い。この環境特性を踏まえ、**未実証を実証と偽って上振れさせない**ことを較正の既定とする。
- **最終確定は単一の grounding owner が担う**: 連鎖 finding の重大度確定は §1.3 の統合タスク（検証ステータスを見られる単一の code-read worker）に集約する。コードを読めない上位層が domain worker の暫定値を無検証で合算しない。

## 4. エビデンス要件

全ての脅威主張にリポジトリパス参照を要求する。タスクの acceptance_criteria に以下を含める:

```
各脅威に対し、該当するソースコードのファイルパスと行番号を明記すること
```

## 5. セキュリティコントロールカテゴリ

緩和策（Step 7）の設計時に参照するカテゴリ:

| カテゴリ | 対象 |
|---------|------|
| ID/アクセス管理 | 認証・認可・RBAC |
| 入力保護 | バリデーション・サニタイズ・パラメータ化クエリ |
| ネットワーク | TLS・ファイアウォール・レート制限 |
| データ保護 | 暗号化・マスキング・保持ポリシー |
| 分離 | サンドボックス・コンテナ・最小権限 |
| 可観測性 | 監査ログ・アラート・異常検知 |
| サプライチェーン | 依存関係スキャン・署名検証・SBOM |
| 変更管理 | コードレビュー・CI/CD・デプロイ承認 |

## 6. アンチパターン

| パターン | 問題 | 改善 |
|---------|------|------|
| **スコープ過大** | 全てを一度に**同じ深さで**分析しようとし網羅性も深さも不足 | スコープ限定は「**分析の深さの優先順位付け**」にのみ適用する。Step 1（§1.1）の機械的列挙はリポジトリ全体（union）を必ずカバーし、**列挙範囲を削ってはならない**。列挙後に優先コンポーネントから段階的に深掘りする |
| **未走査領域の範囲外宣言** | 走査していない層を「リポジトリ外」「対象外」と公理化し、層まるごとが最終レポートから欠落（集約層は復元不能） | §1.1 に従い、範囲外宣言には機械的列挙の否定の根拠（空の grep/ls-files 結果）を必須化 |
| **エビデンスなし** | コード参照なしに脅威を主張 | 全ての脅威に該当ファイルパス・行番号を必須化（§4） |
| **優先度評価スキップ** | 全脅威を同列に扱い対策の優先順位が不明 | §3 のリスク優先度で High/Medium/Low を必ず分類 |
| **資産分類の省略** | 保護対象を明確にせず脅威を列挙 | §2 の資産カテゴリで網羅的に分類してから分析開始 |
| **理論最悪値での重大度上振れ** | 未実証の攻撃連鎖を最悪ケース前提で Critical 認定し、本文では「未実証」と告白する | §3.1 の上限ガード。未実証連鎖は High + `要追加検証` 止まり、Critical は実証済 or 前提条件なし成立のコード確証に限定 |
