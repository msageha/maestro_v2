# Self-Evaluation Patterns

Agent が出力・報告前に実行する自己チェックフレームワーク。

## 1. 完了判定チェックリスト

報告前に順に検証する。

| # | チェック | 判定基準 | 不合格時 |
|---|---------|---------|---------|
| 1 | **Criteria Match** | acceptance_criteria 各条件に証拠あり | 追加作業 or `[未完了]` |
| 2 | **Scope Guard** | 変更が content/constraints 範囲内 | スコープ外変更を revert |
| 3 | **Consistency** | ファイル間で型・命名が整合 | 不整合を修正 |
| 4 | **Build Check** | 検証コマンドが PASS | 修正 or `--status failed` |
| 5 | **Side Effects** | 意図しない変更なし | revert |

```
[作業完了] → [チェック] → [全PASS] → [報告]
                        → [FAIL] → [修正→再チェック（最大2回）] → [修正不能→failed]
```

## 2. 品質メトリクス（3軸）

| 軸 | 定義 | 検証方法 |
|----|------|---------|
| **正確性** | criteria の意図を正しく反映 | 条件を1つずつ照合 |
| **完全性** | 必要な変更に漏れなし | ファイルリストと content を突合 |
| **一貫性** | 既存パターン・命名規則と整合 | 周辺コードと比較 |

## 3. 失敗の早期検知トリガー

以下検知時は作業を継続せず方向転換を判断する。

| トリガー | 兆候 | アクション |
|---------|------|----------|
| **前提崩壊** | 記載されたファイル・関数・型が存在しない | 停止→`failed` 報告 |
| **スコープ膨張** | 変更ファイルが想定の3倍超 | 停止→追加範囲を報告 |
| **循環修正** | 同じエラー修正を2回以上反復 | アプローチ変更 or `failed` |
| **依存ブロック** | 先行タスク成果物が期待と異なる | 差異を報告→`failed` |

## 4. 過信防止: 不確実性レベル

| レベル | 状態 | アクション |
|-------|------|----------|
| **確信** | 証拠に基づく判断 | 通常報告 |
| **推定** | コード読解のみ、実行未確認 | `[注意事項]` に「推定: {根拠}」 |
| **不確実** | 複数解釈が可能 | 保守的解釈 + 代替を `[注意事項]` に列挙 |
| **判断不能** | criteria 自体が矛盾・不十分 | `failed` + 矛盾点を記載 |

## 5. Evaluator-Optimizer パターン

品質要求が高いタスク（`persona_hint: quality-assurance` 等）向け。

```
[実装] → [ルーブリック評価] → [全観点 >= 3] → [完了]
                             → [< 3] → [改善→再評価（最大3回）]
```

| 観点 | 1 (不可) | 3 (許容) | 5 (優秀) |
|------|---------|---------|---------|
| 要件充足 | 半数未満 | 全条件充足 | エッジケース考慮 |
| コード品質 | ビルドエラー | 既存パターン準拠 | 完全準拠+高可読性 |
| テスト | なし | 正常系あり | 正常+異常+境界値 |

## 6. Role別適用

| Role | 推奨チェック |
|------|------------|
| Orchestrator | 曖昧性検出（structured-ai-communication §4） |
| Planner | criteria が SMART か、依存関係漏れなし |
| Worker | 完了判定チェックリスト（§1）全項目 |
