# Context Window Efficiency

## 1. 情報圧縮の原則

### 必要十分な情報量の判断基準

| 含めるべき情報 | 省くべき情報 |
|--------------|-------------|
| 変更の意図（なぜ変更したか） | 実装の詳細手順（どうやって変更したか） |
| 型・API の変更点 | 変更しなかったファイルの説明 |
| 後続タスクが知るべき副作用 | 自明な変更の繰り返し説明 |
| エラー発生時の再現条件 | 成功した試行の過程 |
| 制約・前提条件の変化 | ツール出力の全文コピー |

### 圧縮テクニック

- **差分ベース記述**: 「`User` 型に `Email string` フィールドを追加」のように、変更前後の差分を簡潔に記述する
- **参照ベース記述**: 長い出力は「`go test` で 3 件失敗（TestA, TestB, TestC）」のようにキーポイントのみ記載する
- **構造化記述**: 自由文より表・箇条書きを使い、同じ情報量をより少ない文字数で表現する

## 2. タスク間情報伝達パターン

### Summary の情報密度最適化

`--summary` はタスク結果の唯一の伝達手段である。以下を守る:

| タグ | 記載ルール |
|------|----------|
| `[変更理由]` | 変更の意図 + 主要な変更点を 1-2 行で。ファイル名は変更意図が伝わる場合のみ含める |
| `[注意事項]` | 後続タスクの **行動を変える** 情報のみ。「特になし」は書かない（タグごと省略） |
| `[未完了]` | スコープ外で発見した問題。完了時は `なし` |

**アンチパターン:**
- 変更ファイルの羅列だけの summary（`--files-changed` で報告済み）
- 実装手順の説明（後続タスクには不要）
- 「問題なく完了しました」等の定型文（`--status completed` で表現済み）

### 段階的情報収集パターン

大量の情報が必要な場合、一度に全てを読み込まず段階的に収集する:

1. **スコープ特定**: `Explore` SubAgent で影響範囲のファイルリストを取得
2. **優先読み込み**: 変更対象ファイルのみを先に読む
3. **必要時展開**: 依存先は実際にエラーや不明点が発生した時点で読む

## 3. 大量ファイル変更時の構造化

変更ファイルが 5 件以上の場合、summary を以下の構造で記載する:

```
[変更理由] <目的を1行で>
- カテゴリA: file1, file2（変更内容を簡潔に）
- カテゴリB: file3, file4, file5（変更内容を簡潔に）
[注意事項] <後続への影響>
[未完了] なし
```

カテゴリ例: 型定義 / ハンドラ / テスト / 設定 / マイグレーション

## 4. Compaction Recovery 時の情報保全

コンテキスト圧縮が発生しても作業を継続できるよう、以下の情報を意識的に残す:

### 圧縮前に保全すべき情報

| 優先度 | 情報 | 保全方法 |
|--------|------|---------|
| 必須 | タスク配信フィールド（task_id, command_id 等） | コンテキスト内に残る（システム保証） |
| 必須 | acceptance_criteria の達成状況 | 作業の区切りごとに自己確認する |
| 高 | 変更済みファイルの一覧 | `git diff --name-only` で再取得可能 |
| 高 | 未完了の作業項目 | 複雑なタスクでは中間メモとして出力する |
| 中 | 試行錯誤の結果（何が失敗したか） | learnings として報告候補にする |

### 回復パターン

圧縮後に状況が不明な場合:

1. `git status` / `git diff --stat` で現在の変更状態を確認
2. acceptance_criteria と照合し、達成済み項目を特定
3. 未達成項目から作業を再開（既に達成済みの作業を繰り返さない）

## 5. SubAgent への情報伝達

SubAgent に委譲する際のプロンプトも情報密度を意識する:

- **具体的なファイルパスとシンボル名** を指定する（「関連ファイルを探して」ではなく「`internal/auth/handler.go` の `Login` 関数」）
- **期待する出力形式** を指定する（「影響範囲をリストで返して」）
- **不要な情報の除外** を指示する（「テストファイルは除外」「インターフェース定義のみ」）
