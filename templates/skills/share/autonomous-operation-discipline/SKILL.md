# Autonomous Operation Discipline

長時間の自律運用でエージェントが陥る失敗モード — 裁量が曖昧で routine 作業でも停止する / 複数の「真実」が衝突したまま進む / 誤りが黙って上書きされる / 誤った既成事実が毎ターン再注入される — に対する判断規律。

Maestro が構造で強制している事項（queue/state/dashboard の tracker、3 層委譲、retry lineage、verbatim commit）を prose で再実装するものではない。本 skill は構造で縛れない**判断・裁定・自己点検**の隙間だけを埋め、新たな gate やフロー強制は導入しない。

## 1. 裁量のしきい値 — 聞くべきときと進むべきとき

裁量の境界を曖昧にしたまま運用すると、routine 作業で毎回停止するか、逆に不可逆な判断を勝手に進めるかの両極に振れる。境界を先に固定する:

| 領域                                                                        | 行動                                                                                     |
| --------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| **decide → execute → log**: 可逆で、割り当てスコープ内で、規約に沿った判断  | 実行して記録し、次の報告で伝える。事前に許可を求めて停止しない                           |
| **escalate**: 不可逆な操作、外部公開、スコープの変更、真実の裁定            | 選択肢 + 推奨 + 根拠を添えて上位（Worker→Planner→Orchestrator→ユーザ）に委ねる           |
| **常に明示的肯定が必要**: 削除・公開・課金など取り返しのつかない commitment | 曖昧な返答・沈黙・過去の類似承認から承認を**推論しない**。明示的な肯定だけを承認と見なす |

escalate は失敗ではない。「escalate すべき判断を自力で進めた」ことが失敗である。

## 2. source-of-truth の競合階層

2 つの情報源が矛盾したとき、暗黙に片方を選んで進むとドリフトが始まる。優先順位を固定する:

```
ユーザの明示指定 > 承認済み成果物 (acceptance 済み result・merge 済み変更)
              > 検証済み実データ (実行結果・実ファイル・実測値)
              > 生成コンテンツ (自分や他エージェントの推論・要約)
```

- 同位の真実が衝突したら（例: 並列 Worker の成果が矛盾）、自分で黙って裁定せず、**衝突の内容と裁定案を上位層に提示し、裁定の結果を記録に残す**
- 裁定は記録するだけで成果物を書き換えない（verbatim commit と両立）
- 自分の要約・推論は常に最下位。実ファイル・実行結果と食い違ったら自分の側を訂正する

## 3. サイクル末 self-audit と correction log

各サイクル（タスク完了報告・計画更新・状態遷移）の末尾で、成果物ではなく**前提**を点検する:

- 参照したエンティティ（ファイル・ID・タスク・API）は実在を確認したか、それとも名前から推測したか
- tracker（queue/state/summary）と実状態（git・ファイルシステム・実行結果）がドリフトしていないか
- 割り当てられた constraints を守ったか

誤りを発見したら**隠さず correction として記録する**（summary の `[注意事項]`、訂正の `--learnings`）。誤りの記録は運用記録の正常な一部であって恥ではない。黙って上書きすると、上位層は誤った前提のまま次の判断を行う。

> 完了判定のチェックリストは `self-evaluation-patterns` が担う。本節はその外側 — **事実と前提のドリフト検査**を担い、重複しない。

## 4. 誤った canonical fact は「全 root」で直す

誤った事実（誤ったパス・ID・仕様理解）が永続層に保存されると、会話中で 1 回訂正しても**保存元から毎ターン再注入され、誤りが蘇生し続ける**。訂正するときは:

1. その事実が保存されている root を全て列挙する（instructions・state・learnings・メモリ・ドキュメント）
2. 全 root を同時に修正し、重複エントリを prune する
3. 自分に書込権限のない root（`.maestro/` 制御プレーン等）は、訂正内容を明記して上位層に報告する

「1 箇所直したから解決」は、再注入される root が 1 つでも残っていれば未解決である。

## 5. よくある言い訳（Common Rationalizations）

| 言い訳                                                 | なぜ間違いか                                                                                    |
| ------------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| 「前回似た操作が承認されたので今回も承認済みと見なす」 | 承認は文脈ごと。類似からの推論は §1 の「明示的肯定」原則に反し、不可逆操作で最も事故を起こす    |
| 「矛盾は軽微なので手元の解釈で進める」                 | 軽微な矛盾の暗黙裁定が累積したものがドリフト。§2 の階層で機械的に解決するか、衝突として記録する |
| 「誤りは直したので報告しなくてよい」                   | 上位層は誤った前提で既に判断している可能性がある。correction の記録が無いと巻き戻せない         |
| 「メモリの誤記は次に気づいた人が直すだろう」           | 再注入は自動で、気づく機会は毎ターン失われていく。発見した者が全 root を直すのが最安の時点      |

## 6. 危険信号（Red Flags）

- 同じ訂正を 2 回以上している（→ 直していない root が残っている。§4）
- 「たぶん承認されるはず」という語彙で不可逆操作を進めようとしている
- 実行結果と自分の要約が食い違ったとき、実行結果の側を疑っている
- routine な判断のたびに確認を求めて停止している（→ §1 の decide→execute→log 領域）

## 7. 完了チェックリスト（Verification）

サイクル末・報告前に確認する:

- [ ] 今回の判断のうち escalate すべきもの（不可逆・スコープ変更・真実の裁定）を自力で進めていない
- [ ] 参照した事実は実在確認済みか、未確認のものは「推定」と明示した
- [ ] 発見した誤り・矛盾は correction として記録した（黙った上書きが無い）
- [ ] 訂正した事実について、保存されている全 root を列挙・修正した（または権限外 root を報告した）
