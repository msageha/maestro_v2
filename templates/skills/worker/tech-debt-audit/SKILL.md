# Tech Debt Audit

「codebase health check」「アーキテクチャレビュー」「cleanup 計画の入力」を求められたときの、機械ツール裏付けの技術的負債監査。**全ての主張に file:line の引用を付け、印象論を禁止する** (`source-grounded-response` と同じ精神)。

## 1. 位置付け — レポートであって修正ではない

- 本 skill の成果物は**優先度付き監査レポートのみ**。監査中に発見した負債を skill 内で修正しない (worktree 隔離・スコープ規律を守る)
- 修正は Planner がレポートを入力として別タスクに分解する。レポートの 1 項目 = 分解可能な 1 修正候補になる粒度で書く
- 変更ごとの構造点検は `code-structure-discipline` が担う。本 skill は**一回性の全体監査**であり、検出観点は共有しても目的が違う

## 2. ツール選択と degrade 規約

構文木ベースの検出を優先し、環境に無ければ段階的に degrade する。**どの層で監査したかをレポート冒頭に必ず明記する**:

| 層              | ツール                                                    | 検出できるもの                                |
| --------------- | --------------------------------------------------------- | --------------------------------------------- |
| 構文木 (優先)   | ast-grep / tree-sitter 系 CLI                             | 構文パターン (重複構造・複雑度・抑制コメント) |
| 言語ネイティブ  | コンパイラ警告・lint (go vet / golangci-lint / eslint 等) | 型安全・dead code・複雑度 (言語標準の定義で)  |
| LSP             | 参照列挙・シンボル検索                                    | 未参照シンボル・参照の偏り                    |
| grep (最終手段) | パターン検索                                              | 文字列レベルの兆候のみ (誤検知を明示する)     |

ツールの存在は実行前に確認する (`command -v ast-grep` 等)。構文木ツールが無い場合は「構文木検出をスキップし grep/LSP に degrade した」とレポートに明示し、検出の網羅性が落ちる次元を列挙する。

## 3. 監査次元

各次元を独立に走査する。全次元が必須ではなく、タスクの `content` で範囲が指定されていればそれに従う:

| 次元               | 検出例                                                                             | 主な手段                       |
| ------------------ | ---------------------------------------------------------------------------------- | ------------------------------ |
| 重複               | コピペされた関数・類似構造の並立                                                   | 構文木 / grep                  |
| 密結合・循環依存   | 逆向き参照、モジュール間の相互 import                                              | 言語ネイティブ / LSP           |
| dead code          | 未参照シンボル、コメントアウト残骸、到達不能分岐                                   | lint / LSP                     |
| 複雑度             | 巨大関数・深いネスト・引数過多                                                     | lint / 構文木                  |
| 型安全             | 型/lint 抑制コメント、unchecked cast (`code-craft-anti-patterns` の AP-1/2 の蓄積) | 構文木 / grep                  |
| テスト欠落         | 公開 API でテスト未参照のもの、アサーション無しテスト                              | LSP / grep                     |
| エラーハンドリング | 握りつぶし (`_ = err`・空 catch)、panic 直呼び                                     | 構文木 / grep                  |
| 依存の鮮度         | 非推奨 API 使用、EOL 依存                                                          | 言語ネイティブ / manifest 確認 |
| ドキュメント乖離   | README・コメントと実装の食い違い                                                   | Read + 突合                    |

## 4. レポート様式

成果物は `TECH_DEBT_AUDIT.md` (タスク指示があればそのパス)。各 finding は次の形式:

```
### TD-<連番>: <一言サマリ>
- 次元: <監査次元> / severity: high|medium|low / 工数: S|M|L / confidence: high|medium|low
- 根拠: <file:line> (+ 検出コマンドと出力の要点)
- 影響: <放置した場合に何が起きるか>
- 修正案: <1 タスクに分解可能な粒度で>
```

- **severity** は影響 (障害リスク・変更コスト増) で判定し、**工数** (S: 1 タスク / M: 2-3 タスク / L: 要設計) と独立に付ける
- 優先度は severity × 工数の組で並べる (high×S が最優先、low×L が最後)
- 検出ツールの出力に基づかない「印象」は書かない。確信が持てない finding は confidence: low として残し、削らない (絞り込みは Planner の工程)

## 5. よくある言い訳 (Common Rationalizations)

| 言い訳                                                  | なぜ間違いか                                                                                    |
| ------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| 「明らかな負債なので引用は省略してよい」                | 引用の無い finding は Planner が検証できず、タスク分解の入力にならない。引用が無いなら書かない  |
| 「ついでに簡単な負債は直しておく」                      | 監査タスクの成果物はレポート。混ぜた修正は diff を汚し、並列 worker との conflict 源になる      |
| 「重要そうな finding だけに絞って報告する」             | 絞り込みは Planner の工程。監査工程での省略は「検出されなかった」と区別が付かず網羅性を破壊する |
| 「ast-grep が無いので複雑度の監査は黙ってスキップする」 | スキップ自体は正しいが、黙るのが誤り。何を検出できていないかはレポートの一部である              |

## 6. 危険信号 (Red Flags)

- file:line の無い段落を書いている
- 監査中に Edit / Write でソースコードに触れている
- ツールの出力を確認せずに「よくあるパターンだから存在するはず」で finding を起こしている
- レポート冒頭に使用ツールと degrade の有無を書いていない

## 7. 完了チェックリスト (Verification)

- [ ] 全 finding に file:line と検出手段の記録がある
- [ ] レポート冒頭に使用ツール・スキップした次元・degrade の有無を明記した
- [ ] ソースコードへの変更が無い (`git status --porcelain` がレポートファイル以外空)
- [ ] 各 finding の修正案が 1 タスクに分解可能な粒度になっている
- [ ] severity / 工数 / confidence が全 finding に付与され、優先度順に並んでいる
