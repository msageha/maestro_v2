# Resilient Execution

エラー発生時のリトライ判断・バックオフ戦略・段階的エスカレーションを体系化する。

## 1. エラー分類と対応方針

| 分類 | リトライ可否 | 戦略 | 例 |
|------|------------|------|-----|
| **Transient** | 可 | 自動リトライ | ネットワークタイムアウト、一時的503 |
| **Recoverable** | 条件付き | 修正後リトライ | 設定ミス、パス誤り、権限不足 |
| **Permanent** | 不可 | 即時エスカレーション | 型エラー、ロジックバグ、仕様不一致 |
| **Partial** | 部分的 | 成功部分を保持 | 3ファイル中2ファイル成功 |

## 2. リトライ戦略

```
エラー発生
  ├─ Permanent? → 即時 failed 報告（--no-retry-safe）
  ├─ Transient? → リトライループへ
  ├─ Recoverable? → 修正 → リトライ
  └─ Partial? → 成功部分を保持 → 残りをリトライ or 報告（--partial-changes）
```

### リトライ上限

| 操作種別 | 最大リトライ | 待機戦略 |
|---------|------------|---------|
| 外部 API 呼出 | 3回 | 指数バックオフ（1s→2s→4s） |
| Bash コマンド | 2回 | 即時（原因修正後） |
| SubAgent 委譲 | 1回 | 直接実行に切替 |
| ビルド/テスト | 2回 | 即時（修正後） |

## 3. 段階的エスカレーション

修復試行が失敗した場合、段階的に対応を拡大する。

```
[Level 1: 自力修復]
  │ 同一アプローチで修正 → 成功？ → 完了
  │                      → 失敗
  ▼
[Level 2: アプローチ変更]
  │ 別手法で試行 → 成功？ → 完了
  │               → 失敗
  ▼
[Level 3: 失敗報告]
  │ --status failed + 試行内容を summary に記載
  │ 判断: --no-retry-safe or デフォルト（retry可）
```

| Level | アクション | 例 |
|-------|----------|-----|
| L1 | 直接修正 | typo修正、import追加、パス修正 |
| L2 | 代替手法 | 別API使用、別ライブラリ、手動実装 |
| L3 | 報告 | 原因・試行内容・推奨対応を記載 |

## 4. 回復パターン

| シナリオ | 回復手法 |
|---------|---------|
| コマンドタイムアウト | タイムアウト値拡大→リトライ |
| SubAgent 失敗 | 直接実行に切替 |
| テスト不安定 | 単独実行→環境依存確認 |

## 5. 報告時の判断基準

| 状態 | フラグ | 理由 |
|------|--------|------|
| ロジックバグで修正不能 | `--no-retry-safe` | 同入力で同結果 |
| 環境要因で一時失敗 | デフォルト | リトライで成功見込み |
| 2/3ファイル変更済み | `--partial-changes` | 中途半端な状態 |
| 変更なしで失敗 | デフォルト | クリーンな状態 |

## 6. アンチパターン

| パターン | 問題 | 改善 |
|---------|------|------|
| **盲目リトライ** | Permanent を繰り返す | 分類判断を先行 |
| **即時放棄** | Transient で即 failed | リトライ上限まで試行 |
| **無限ループ** | 上限なしリトライ | 上限を厳守 |
| **状態無視** | partial-changes 未報告 | 変更状態を正確に報告 |
