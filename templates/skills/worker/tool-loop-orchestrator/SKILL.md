# Tool Loop Orchestrator

複数ツールを組み合わせた逐次実行ワークフローの設計・制御パターン集。

## 1. 基本ループ構造

```
[調査] → [実装] → [検証] → [修正?] → [報告]
  │         │        │        │
  │         │        ├─ PASS → 報告
  │         │        └─ FAIL → 修正 → 再検証（最大N回）
  │         │                           └─ 超過 → failed報告
  │         └─ エラー → エラー診断 → 修正 or failed
  └─ 前提不成立 → failed報告
```

## 2. ステップ定義テンプレート

各ステップを以下の属性で定義し、実行順序を管理する:

| 属性 | 説明 | 例 |
|------|------|-----|
| **目的** | このステップで達成すること | 影響範囲の特定 |
| **ツール** | 使用するツール/SubAgent | Explore SubAgent |
| **入力** | 必要な情報 | 対象ファイルパス |
| **成功条件** | 次ステップに進む条件 | ファイルリスト取得済み |
| **失敗時** | 失敗時のアクション | failed報告 / 代替手段 |

## 3. 典型的ワークフローパターン

### パターンA: 調査→実装→検証

| Step | 操作 | ツール | 失敗時 |
|------|------|--------|--------|
| 1 | 影響範囲調査 | Explore SubAgent | failed報告 |
| 2 | ファイル編集 | Edit/Write | failed報告 |
| 3 | 基本検証 | Bash（ビルド/lint） | Step 2に戻り修正 |
| 4 | テスト実行 | Bash（テスト） | Step 2に戻り修正 |

### パターンB: 複数ファイル逐次変更

| Step | 操作 | 注意 |
|------|------|------|
| 1 | 型定義・インターフェース変更 | 依存元を先に変更 |
| 2 | 実装コード変更 | Step 1の型に合わせる |
| 3 | テストコード変更 | Step 2の実装に合わせる |
| 4 | 全体検証 | ビルド+テスト |

## 4. 停止条件の判定

ループを停止すべきタイミング:

| 条件 | アクション |
|------|----------|
| 検証PASS | completed報告 |
| 同一エラーが2回連続 | アプローチ変更 or failed |
| 修正ループ3回超過 | failed報告（循環修正） |
| スコープ膨張（変更ファイル想定の3倍超） | 停止→failed報告 |
| 前提条件の崩壊（ファイル・関数不在） | 即時failed報告 |

## 5. 並列実行の安全規則

SubAgent を並列実行する場合の制約:

| 許可 | 禁止 |
|------|------|
| 異なるファイルの同時編集 | 同一ファイルの同時編集 |
| 読み取り専用の並列実行 | 書き込み+読み取りの競合 |
| 独立したテストの並列実行 | 依存関係のあるステップの並列化 |

## 6. 状態追跡

ループ中の進捗を把握するため、各ステップ完了時に以下を確認する:

- 変更済みファイル一覧（`git diff --name-only` で確認可能）
- 残りのステップ数
- 発生したエラーと対応結果
