# Constraint-Aware Implementation

タスクの制約を明示的に抽出し、実装後に遵守を検証するフレームワーク。

## 1. 制約抽出フロー

タスク受信時に以下のソースから制約を列挙する:

```
[タスク受信]
  ├─ constraints フィールド → 明示的制約
  ├─ acceptance_criteria → 暗黙の制約
  ├─ content の記述 → 文脈的制約
  └─ 既存コードの規約 → 環境的制約
```

## 2. 制約分類体系

| カテゴリ | 例 | 検証方法 |
|---------|-----|---------|
| **性能** | レスポンスタイム、メモリ使用量、N+1禁止 | ベンチマーク、コード確認 |
| **セキュリティ** | 入力バリデーション、認証必須、SQLインジェクション対策 | コードレビュー、パターン検索 |
| **スタイル** | 命名規則、ディレクトリ構成、エラーハンドリング規約 | 既存コードとの比較 |
| **互換性** | API後方互換、既存テスト維持、型シグネチャ保持 | テスト実行、型チェック |
| **スコープ** | 変更対象の限定、特定ファイル以外の変更禁止 | git diff で確認 |

## 3. 制約チェックリスト

実装完了後、報告前に以下を検証する:

| # | チェック | 判定 | 不合格時 |
|---|---------|------|---------|
| 1 | 明示的制約を全て列挙したか | ✅/❌ | 再抽出 |
| 2 | 各制約に対する実装対応があるか | ✅/❌ | 追加実装 |
| 3 | 制約違反がないか（コード確認） | ✅/❌ | 修正 |
| 4 | 既存テストが通るか | ✅/❌ | 修正 |
| 5 | スコープ外の変更がないか | ✅/❌ | revert |

## 4. 曖昧な制約の解決

| 曖昧パターン | 例 | 対応 |
|-------------|-----|------|
| 程度の指定なし | 「高速に」 | 既存実装と同等以上を基準とする |
| 条件の欠落 | 「安全に」 | 入力バリデーション+エラーハンドリングを最低限とする |
| 範囲の不明確 | 「関連コード」 | 直接参照のみに限定し `[注意事項]` に明記 |
| 矛盾する制約 | 性能 vs 安全性 | 安全性を優先し `[注意事項]` にトレードオフを記載 |

優先順位: **セキュリティ > 互換性 > 正確性 > 性能 > スタイル**

## 5. 制約遵守の報告

`--summary` に制約の遵守状況を含める:

```
[変更理由] {実装内容}
[注意事項]
- 制約: {制約1} → 遵守（{根拠}）
- 制約: {制約2} → 遵守（{根拠}）
- トレードオフ: {判断内容}（{理由}）
[未完了] なし
```

## 6. アンチパターン

| パターン | 問題 | 改善 |
|---------|------|------|
| **制約の暗黙仮定** | 抽出せず実装開始 | §1のフローで先に列挙 |
| **検証スキップ** | 「大丈夫だろう」 | §3のチェックリスト実行 |
| **過剰な制約追加** | タスクにない制約を自己追加 | 明示的制約のみ遵守 |
| **制約違反の黙殺** | 報告せず完了 | `[注意事項]` に明記 |
