# Code Craft Anti-Patterns

Worker が生成するコードの品質を守るための、言語横断の craft anti-pattern リスト。BLOCKING = 「これを含む成果物を completed として報告してはならない」という自己規律であり、検出したら修正してから報告する。

## 1. 位置付け（安全性 Tier 1 と混同しない）

- `maestro.md` の Tier 1 禁止（`.maestro/` 直書き・tmux 操作・破壊コマンド等）は**オーケストレーション安全性**の制約で、L1/L2 hook が技術的に強制する。
- 本リストは**コード品質（craft）**の規律で、enforcement は advisory。機械強制は対象プロジェクトの lint / 型チェッカが担い、本 skill はそれを**回避しない**ことを規律化する。
- 言語固有のルール（命名・ファイル構成の詳細等）は対象プロジェクトの lint 設定・既存規約に従う。本リストは言語を問わず成立する原則のみを扱う。

## 2. BLOCKING anti-patterns と Reality

| #    | Anti-pattern                                                                                                   | Reality（なぜダメか）                                                                                                                  |
| ---- | -------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| AP-1 | **型チェック抑制**: `as any` / `@ts-ignore` / `# type: ignore` / unchecked cast 等で型エラーを黙らせる         | 型エラーは設計の矛盾を指す信号。抑制はバグを実行時に先送りし、後続 Worker が原因箇所を特定できなくなる                                 |
| AP-2 | **lint 抑制の無断追加**: `//nolint` / `eslint-disable` / `# noqa` 等をエラー回避のためだけに挿入する           | lint ルールはプロジェクトの合意。個別タスクの都合で穴を開けると、抑制コメントが増殖して lint が形骸化する                              |
| AP-3 | **失敗テストの削除・skip による緑化**: テストを消す / `skip` / `xit` / `t.Skip` に変えてビルドを通す           | テストは仕様の実行可能な記録。消して得た緑は「壊れたことを検知できなくなった」だけで、修正すべきはコードである                         |
| AP-4 | **アサーションの希釈**: 期待値を実装の現在の出力に合わせて弱める・等値比較を「含む」比較に緩める               | AP-3 の変種。テストが実装の追認装置になり、リグレッション検知能力を失う                                                                |
| AP-5 | **空 catch / エラー握りつぶし**: `catch(e) {}`、`_ = err`、例外を握って正常系を装う                            | 失敗が観測不能になり、障害時に「エラーは出ていないのに動かない」最悪のデバッグ体験を生む。処理できない例外は伝播させる                 |
| AP-6 | **catch-all ファイルの新設・肥大化**: `utils` / `helpers` / `misc` / `common` に分類不能なコードを投げ込む     | 「どこにも属さない」は設計の未決定。catch-all は依存の吹き溜まりになり、後続の変更影響分析を不可能にする。責務に沿ったモジュールに置く |
| AP-7 | **エントリポイントへのロジック投入**: `index` / barrel / `__init__` / パッケージルートにビジネスロジックを書く | エントリポイントは re-export と配線の場所。ロジックが混ざると import 順序への暗黙依存と循環参照の温床になる                            |
| AP-8 | **dead code のコメントアウト残置**: 使わなくなったコードをコメントアウトで「念のため」残す                     | 履歴は git が持っている。コメントアウト残置は読み手に「これは復活するのか」という無駄な判断を毎回強い、grep 結果を汚染する             |

## 3. 例外の扱い

抑制・逸脱が本当に必要な場面（外部ライブラリの型定義の欠陥、既知の lint false positive 等）は存在する。その場合は:

1. 抑制箇所に**理由を書いたコメント**を必ず添える（何が壊れていて、なぜ抑制が唯一の手か）
2. 結果報告の `--summary` の `[注意事項]` に抑制の事実と理由を記載する
3. タスクの `constraints` や対象プロジェクトの既存規約が明示的に許容している場合はそちらに従う

黙って抑制することだけが禁止であり、説明責任を伴う抑制は許容される。

## 4. よくある言い訳（Common Rationalizations）

| 言い訳                                                     | なぜ間違いか                                                                                                 |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| 「このテストは今回の変更と無関係に落ちている。消してよい」 | 無関係なら触る必要もない。落ちる原因の調査結果を `[注意事項]` で報告するのが正しい経路。削除はスコープ外変更 |
| 「型エラーの修正は工数がかかる。`any` で先に進める」       | 先に進んだ分のコードは間違った型の上に積まれる。後続タスクの手戻りは抑制で稼いだ時間より必ず大きい           |
| 「エラーは起きないはずの経路なので catch は空でよい」      | 「起きないはず」が起きたときに観測できないのが空 catch。起きないなら panic/raise で fail-fast にする         |
| 「置き場所に迷ったので一旦 utils に入れる。後で移す」      | 「後で」を実行するタスクは存在しない。迷った時点で責務を決めるのが実装の一部                                 |

## 5. 危険信号（Red Flags）

作業中に以下の兆候があれば、anti-pattern に踏み込みかけている:

- ビルド・テストを緑にする手段として「コードを直す」以外の選択肢（削除・skip・抑制）を検討し始めた
- 抑制コメントを書いているのに、理由コメントを書いていない
- 「とりあえず」「一旦」「後で直す」という語彙で自分の変更を説明している
- テストの期待値を、テストを読まずに実装の出力コピペで書き換えている

## 6. 完了チェックリスト（Verification）

completed 報告の前に検証する:

- [ ] `git diff` に新規の型/lint 抑制（`as any` / `nolint` / `noqa` / `ts-ignore` 等）が含まれていない。含まれる場合は理由コメントと `[注意事項]` 記載がある
- [ ] テストの削除・skip 化・アサーション弱体化を行っていない。行った場合はタスクの明示指示に基づく
- [ ] 追加した catch / error handling に握りつぶしがない（ログ・伝播・fail-fast のいずれかで観測可能）
- [ ] 対象プロジェクトの lint / 型チェックを実行し、抑制なしで通過した（コマンドと結果を summary に記録）
