# Source-Grounded Response

報告・分析結果を事実に基づいて構成し、ハルシネーションを防止するパターン集。

## 1. エビデンスレベル

全ての主張に以下のレベルを適用する:

| レベル | 根拠 | 表記 | 例 |
|--------|------|------|-----|
| **確認済み** | コード実行・テスト結果で検証 | そのまま記述 | 「テスト全件PASS」 |
| **参照済み** | ソースコード読解で確認 | ファイルパス+行番号を付記 | 「`handler.go:42` で nil チェックなし」 |
| **推定** | コード読解のみ、実行未確認 | 「推定:」を付記 | 「推定: この関数は呼び出し元が1箇所」 |
| **不確実** | 複数解釈が可能 | 「不確実:」+ 代替を列挙 | 「不確実: キャッシュ or DB直接参照」 |

## 2. 引用ルール

### 必須引用パターン

| 主張の種類 | 必要な引用 |
|-----------|----------|
| 「この関数は〜する」 | ファイルパス:行番号 |
| 「この型は〜を持つ」 | 型定義のファイルパス |
| 「影響範囲は〜件」 | Grep/Glob の検索結果 |
| 「テストが〜」 | テスト実行の出力 |
| 「依存関係が〜」 | import文 or 設定ファイルの参照 |

### 禁止パターン

| パターン | 問題 | 改善 |
|---------|------|------|
| 「おそらく〜でしょう」（根拠なし） | 推測を事実として提示 | 不確実レベルを明示 |
| 「一般的に〜です」 | プロジェクト固有の事実と混同 | コードベースの実態を確認 |
| 自己参照による検証 | 自分の推論を根拠に再利用 | 独立したソースで検証 |

## 3. 循環検証の防止

```
主張A → 根拠は？ → 主張B → 根拠は？ → 主張A（循環！）
```

防止策:
- 全ての根拠チェーンの終端は **外部ソース**（コード、実行結果、ドキュメント）であること
- 自分の推論結果を別の推論の根拠にしない
- 「§Xで述べた通り」は参照であり根拠ではない

## 4. summary報告への適用

`--summary` の各タグに適用する:

| タグ | グラウンディング要件 |
|------|-------------------|
| `[変更理由]` | 変更対象のファイルパスを含める |
| `[注意事項]` | 不確実な点は「推定:」を付記 |
| `[未完了]` | 未完了の理由を具体的に記載（「時間不足」ではなく「〜エラーが解消できず」） |

## 5. 調査報告の構造

researcher ペルソナでの調査結果報告テンプレート:

```
[変更理由] <調査目的と範囲>
[注意事項]
- 確認済み: <検証済みの事実（パス参照付き）>
- 推定: <実行未確認だがコード読解で確認した事項>
- 不確実: <複数解釈がある点と各候補>
[未完了] <調査できなかった範囲と理由>
```

## 6. 判断不能時の対応

根拠が不十分で判断できない場合:

| 状況 | アクション |
|------|----------|
| ファイルが見つからない | `failed` + 探索した範囲を報告 |
| コードが複雑で確信が持てない | 「不確実:」レベルで報告 + 代替解釈を列挙 |
| 外部依存で確認不可 | 確認不可の旨を明記し、確認可能な範囲のみ報告 |
