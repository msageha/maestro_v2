# CI/CD Pipelines

CI/CD パイプライン (ワークフロー定義、quality gate、ビルド・テスト・デプロイの自動化) を**新設・改修する**タスクで使う。パイプラインは「壊れたときに全員の作業が止まるコード」であり、アプリコードと同等以上の規律で扱う。

## 1. 適用条件と棲み分け

適用するのは、CI/CD 定義ファイル (GitHub Actions / GitLab CI / Jenkins 等) や、そこから呼ばれる自動化スクリプトの変更を含むタスク。

| 本 skill が扱わないもの                  | 委ね先                                                     |
| ---------------------------------------- | ---------------------------------------------------------- |
| CI で落ちたテスト・ビルドの原因分析      | `error-diagnosis-patterns`                                 |
| リリース工程 (rollout / rollback) の設計 | `shipping-and-release` (本 skill はそこへ至る自動化の実装) |
| 複数 CLI の逐次実行・状態管理の一般論    | `tool-loop-orchestrator`                                   |
| リトライ戦略の一般論                     | `resilient-execution`                                      |

CI プラットフォーム固有機能 (専用 action / orb / plugin) は既存パイプラインが使っているものに従う。プラットフォームを特定できない・実行環境で CI を起動できない場合は、**パイプラインが呼ぶコマンド列をローカルで直接実行して検証する**ところまで degrade し、「CI 上での実走は未確認」と `[注意事項]` に明記する。

## 2. 設計原則

1. **ロジックはスクリプトへ、YAML は配線だけ**: 条件分岐・複数コマンドの合成は versioned なスクリプト (`scripts/` 等) に置き、パイプライン定義からは呼ぶだけにする。ローカルで同じものを実行できることが再現性の土台になる。
2. **fail-fast に並べる**: 安い検査 (lint / format / 型チェック) を先に、高いもの (統合テスト / E2E / ビルド成果物) を後に。独立なジョブは並列にする。
3. **再現可能にする**: ツールチェーン・依存のバージョンを固定 (lockfile / バージョン指定) する。「その日の latest」に依存するステップは、いつか原因不明で壊れる。
4. **quality gate は明確な合否で**: gate (テスト・カバレッジ・lint・脆弱性スキャン) は閾値と対象を明示し、恒常的に無視される警告を作らない。gate を追加するタスクでは、既存コードが gate を通ることを先に確認する (通らない gate の追加は全員をブロックする)。
5. **シークレットはプラットフォームの秘匿機構で**: 定義ファイル・スクリプト・ログにシークレットを直書き・echo しない。fork からの PR で走るジョブにシークレットを露出しない。
6. **書き込み系ステップは冪等に**: deploy・publish・タグ付けは再実行が安全になるように書く (既存チェック、`--if-not-exists` 相当、上書き方針の明示)。
7. **最小権限**: パイプラインに与える token / 権限はジョブが必要とする範囲に絞る (例: read しかしないジョブに write 権限を与えない)。

## 3. 変更時の手順

1. 既存パイプラインの構造 (ジョブ・依存関係・トリガー・キャッシュ・シークレット参照) を読み、変更の影響範囲を把握する。
2. パイプラインが呼ぶコマンド列を**まずローカル (worktree) で実行**して通す。CI 上のトライ&エラーで検証を始めない。
3. 定義ファイルの構文検証を行う (プラットフォームの validator / actionlint 等があれば使い、無ければ YAML parse 確認まで)。
4. トリガー条件 (branch / path / event) の変更は、意図しないジョブの発火・停止を招きやすい。変更前後で「いつ走るか」を表にして summary に書く。
5. キャッシュを触る場合、キーの構成 (lockfile hash 等) と無効化条件を明記する。「キャッシュが原因の不可解な失敗」は最頻出の CI 事故。

## 4. アンチパターン (禁止)

- **緑にするための緩和**: 落ちるテストの skip 化、gate 閾値の引き下げ、`continue-on-error` の安易な付与で CI を通す
- **巨大単一ジョブ**: 全部を 1 ジョブに直列で詰め、失敗位置も並列性も失う
- **定義ファイル内の秘密**: トークン・パスワードのハードコード、デバッグ目的の env dump
- **未検証 push の繰り返し**: ローカル検証せず CI を試行錯誤の実行環境として使う (履歴と計算資源を汚す)
- **消せない失敗の放置**: flaky なステップを retry で包んで恒久化する。flaky の原因調査を別タスクとして起票する方が正しい

## 5. よくある言い訳 (Common Rationalizations)

| 言い訳                                                | なぜ間違いか                                                                                   |
| ----------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| 「CI でしか動かない処理なので push して試すしかない」 | 大半のステップはコマンド列であり、ローカル実行で 9 割検証できる。CI 固有部だけを CI で確認する |
| 「このテストは flaky なので skip して gate を通す」   | skip は gate の意味を静かに殺す。skip するなら理由・期限・課題化をセットで summary に残す      |
| 「とりあえず admin token を使えば権限エラーは出ない」 | 過剰権限のパイプラインは供給網攻撃の標的になる。権限エラーは必要権限を特定する情報である       |
| 「latest を使えば常に最新でメンテ不要」               | 破壊的変更が来た日にパイプラインが原因不明で赤くなる。更新は明示的な diff として行う           |
| 「デバッグのために env を全部出力した。あとで消す」   | ログは残る。シークレットが載った時点でローテーションが必要になる。最初から出さない             |

## 6. 危険信号 (Red Flags)

- 定義ファイルの diff に対して、呼び出されるコマンドを一度もローカル実行していない
- 「CI を通すため」にアプリコードでなくパイプライン側の検査を弱める編集をしている
- シークレット参照を echo / printenv しようとしている
- トリガー条件を変えたのに、どのイベントで走る/走らなくなるかを確認していない
- 再実行すると二重 publish・二重 deploy になるステップをそのまま追加している

## 7. 完了チェックリスト (Verification)

- [ ] パイプラインが実行するコマンド列をローカルで実行し、結果 (コマンド + 出力要点) を summary に記録した
- [ ] 定義ファイルの構文検証を行った (使ったツール、無い場合はその旨)
- [ ] トリガー・ジョブ依存関係の変更内容を「変更前 → 変更後」で明記した
- [ ] シークレットの直書き・ログ出力が無いことを diff 全体で確認した
- [ ] gate を追加・変更した場合、現行コードベースがそれを通ること (または通らない場合の扱い) を確認・記録した
