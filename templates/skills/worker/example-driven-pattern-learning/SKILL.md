# Example-Driven Pattern Learning

Few-shot 例示を活用し、コード生成・テスト・報告の品質と一貫性を高めるメタスキル。

## 1. 例示駆動フロー

```
[パターン抽出] → [Good/Bad対比作成] → [テンプレート化] → [適用] → [一貫性検証]
                                                          → [不一致→パターン修正→再適用]
```

## 2. パターン抽出の3層構造

既存コードや指示から段階的にパターンを抽出する。

| 層 | 対象 | 抽出方法 | 例 |
|----|------|---------|-----|
| **原則層** | 守るべきルール | constraints・規約から | `エラーは必ずラップして返す` |
| **パターン層** | 具体的な実装パターン | 既存コードから | `fmt.Errorf("xxx: %w", err)` |
| **例示層** | Good/Bad の対比 | パターン層から導出 | Bad: `return err` / Good: `return fmt.Errorf(...)` |

## 3. Good/Bad 対比テンプレート

一貫性を確保するため、必ず対比形式で例示を構成する。

| 要素 | 内容 |
|------|------|
| **コンテキスト** | どの場面で適用するか（1文） |
| **Bad** | 避けるべきパターン + 問題点 |
| **Good** | 推奨パターン + 理由 |
| **判定基準** | Good/Bad を区別する具体的条件 |

## 4. 適用場面と例示数

| 場面 | 推奨例示数 | 目的 |
|------|-----------|------|
| コード生成 | 2-3 | スタイル・命名の一貫性 |
| テストパターン | 3（正常/境界/異常） | ケース網羅の型を示す |
| summary 報告 | 1 | タグ構造の統一 |
| エラーハンドリング | 2 | ラップ・分類の統一 |

## 5. 一貫性検証

| チェック | 方法 | 不合格時 |
|---------|------|---------|
| **命名規則** | 既存コードの命名と比較 | パターン層を再確認し修正 |
| **構造パターン** | Good 例との構造一致 | 例示に立ち戻り再生成 |
| **エラー形式** | プロジェクト規約と照合 | 規約準拠に修正 |

## 6. アンチパターン

| パターン | 問題 | 改善 |
|---------|------|------|
| **例示なし生成** | スタイル不統一 | 最低1つの Good 例を参照 |
| **例示過多** | コンテキスト圧迫 | 場面あたり最大3例 |
| **Bad のみ提示** | 正解が不明 | 必ず Good を対にする |
| **古い例示固執** | 現行規約と乖離 | 周辺コードから最新パターンを再抽出 |
