# Evaluation-Driven Quality

体系的な評価基準に基づく反復的品質改善サイクルを定義する。

## 1. 評価フレームワーク

```
[Input: タスク要件] → [Output: 成果物] → [評価] → [全軸 >= 3] → [完了]
                                           → [< 3] → [改善→再評価（最大3回）]
                                               → [3回失敗→現状ベストで報告]
```

## 2. 評価軸とルーブリック

| 軸 | 1 (不可) | 2 (不足) | 3 (許容) | 4 (良好) | 5 (優秀) |
|----|---------|---------|---------|---------|---------|
| **正確性** | 要件と無関係 | 半数未達 | 全条件充足 | エッジケース考慮 | 堅牢なエラー処理 |
| **完全性** | 大幅な欠損 | 主要部分のみ | 漏れなし | テスト付き | テスト+文書 |
| **一貫性** | 規則無視 | 部分準拠 | 既存パターン準拠 | 高い可読性 | 模範的コード |

合格基準: **全軸 3 以上**（許容レベル）

## 3. 段階的検証サイクル

| Stage | 検証内容 | 手法 | 不合格時 |
|-------|---------|------|---------|
| S1 **要件照合** | acceptance_criteria との突合 | 条件を1つずつ確認 | 未充足項目を実装 |
| S2 **構造検証** | ビルド通過・構文エラーなし | ビルドコマンド実行 | エラー修正 |
| S3 **動作検証** | テスト PASS | テストコマンド実行 | テスト失敗箇所を修正 |
| S4 **整合検証** | 既存コードとの一貫性 | 周辺コード比較 | パターン統一 |

```
[S1] → PASS → [S2] → PASS → [S3] → PASS → [S4] → PASS → [完了]
  ↑              ↑              ↑              ↑
  └── FAIL ──────┘── FAIL ──────┘── FAIL ──────┘── [修正→該当Stageから再開]
```

## 4. Severity 分類と対応優先度

| Severity | 定義 | 対応 | 例 |
|----------|------|------|-----|
| **Critical** | 機能不全・ビルド不可 | 即時修正必須 | コンパイルエラー、nil参照 |
| **High** | 主要機能の欠陥 | 報告前に修正 | テスト失敗、型不一致 |
| **Medium** | 品質基準未達 | 可能なら修正 | 命名規則違反、冗長コード |
| **Low** | 軽微な改善余地 | summary に記載 | コメント不足、フォーマット |

対応: Critical/Highは報告前に修正必須。Mediumは可能なら修正。Lowは`[注意事項]`に記載のみ。

## 5. 改善ループの制御

| 状態 | アクション |
|------|----------|
| 1回目で全軸 >= 3 | 完了報告 |
| 2回目で改善しない | アプローチ変更 |
| 3回目でも未達 | 現状ベスト+`[未完了]`で報告 |
| 別軸が悪化 | 悪化を優先修正 |

## 6. アンチパターン

| パターン | 問題 | 改善 |
|---------|------|------|
| **主観評価** | 「良さそう」で完了 | ルーブリックで判定 |
| **過剰品質** | 全軸5を追求 | 全軸3で十分、スコープ厳守 |
| **検証スキップ** | テスト未実行で報告 | S1-S4 を順に実行 |
| **改善固執** | 無限改善ループ | 最大3回で打ち切り |
