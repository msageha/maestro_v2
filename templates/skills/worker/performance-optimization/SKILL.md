# Performance Optimization

「遅い」「重い」「性能を改善する」タスクで使う。原則はひとつ: **計測していない最適化は最適化ではない**。プロファイルで特定したボトルネックだけを直し、前後比較で効果を証明する。

## 1. 適用条件と棲み分け

適用するのは、性能改善・性能要件達成・性能劣化調査を目的とするタスク。

| 本 skill が扱わないもの                | 委ね先                                                                |
| -------------------------------------- | --------------------------------------------------------------------- |
| 常設のメトリクス・ログ基盤の実装       | `observability` (本 skill は改善サイクルの中の一時的・比較目的の計測) |
| 機能バグ・エラーの原因分析             | `error-diagnosis-patterns`                                            |
| 性能以外も含む制約全般の抽出と遵守検証 | `constraint-aware-implementation` (性能目標値の管理はそちらに従う)    |
| コード全体の負債監査                   | `tech-debt-audit` (性能はその 1 次元にすぎない)                       |

計測ツールは言語ネイティブを最優先する (例: Go は `pprof`/`go test -bench`、JVM は JFR、Python は `cProfile`、Node は `--cpu-prof`、DB は `EXPLAIN`)。プロファイラが使えない環境では、**区間タイマー (処理前後の時刻差) + 件数の対数出力**まで degrade し、分解能の限界を `[注意事項]` に明記する。Web frontend の場合、Core Web Vitals (LCP / INP / CLS) を指標とし、Lighthouse / ブラウザ devtools が使えなければ計測不能として報告する。

## 2. 手順

1. **目標を数値で固定する**: acceptance_criteria から「何を・どの条件で・どこまで」(例: p95 レイテンシ 200ms 以下、N=10^6 で 5 秒以内) を抽出する。数値が無ければ現状値を測って基準線とし、目標が未定義であることを `[注意事項]` に書く。
2. **再現可能な計測系を作る**: 同一入力・同一環境で繰り返せる計測手順 (ベンチマーク、負荷スクリプト、固定データセット) を先に用意する。1 回きりの体感計測を根拠にしない。
3. **基準線を計測する**: 変更前の数値を複数回計測し、ばらつきごと記録する (中央値 + 最悪値)。
4. **プロファイルでボトルネックを特定する**: CPU / メモリ / I/O / 待ち (ロック・ネットワーク) のどれが支配的かをプロファイルで確定させる。上位 1〜3 箇所だけを対象にする。
5. **一度に 1 つ変更し、再計測する**: 複数の最適化を混ぜると効果の帰属が分からなくなる。変更ごとに同じ計測系で前後比較する。
6. **正しさを検証する**: 最適化は挙動を変えやすい (キャッシュの鮮度、並行化の順序、丸め)。既存テストを全部回し、必要なら等価性テストを足す。
7. **やめどきを判断する**: 目標達成、または改善が計測誤差に埋もれたら止める。さらなる改善案は実装せず `[注意事項]` に提案として残す。

## 3. 判断基準: 定番ボトルネックの当たり先

| 症状                         | 最初に疑う場所                                                     |
| ---------------------------- | ------------------------------------------------------------------ |
| データ量に比例して急激に劣化 | 計算量 (ネストループ、N+1 クエリ、線形探索の繰り返し)              |
| CPU は暇なのに遅い           | I/O 待ち・逐次化された外部呼び出し・ロック競合                     |
| メモリが伸び続ける           | 無制限キャッシュ、リスナー/goroutine リーク、巨大中間データの保持  |
| 初回だけ遅い                 | コールドスタート・接続確立・JIT。目標がどちらの状態か確認する      |
| Web で操作がもたつく         | メインスレッド長タスク、過剰な再レンダリング、未分割の巨大バンドル |

アルゴリズム改善 (計算量の次数を下げる) > バッチ化・キャッシュ > マイクロ最適化、の順で検討する。マイクロ最適化はプロファイルが支配的と示した場合のみ。

## 4. アンチパターン (禁止)

- **推測最適化**: プロファイルを取らずに「ここが遅いはず」で書き換える
- **可読性の先払い**: ボトルネックでもないコードを、性能を理由に難読化する
- **キャッシュで蓋**: 根本原因 (N+1 等) を放置してキャッシュを被せる。無効化バグの温床になる
- **ベンチマークの都合に合わせた実装**: 計測ケースだけ速く、実データで遅い最適化
- **正しさの検証省略**: 前後比較の数値だけ見てテストを回さない

## 5. よくある言い訳 (Common Rationalizations)

| 言い訳                                       | なぜ間違いか                                                                     |
| -------------------------------------------- | -------------------------------------------------------------------------------- |
| 「明らかに遅そうな箇所なので計測は不要」     | 支配項の直感は高頻度で外れる。数分の計測が数時間の無駄な最適化を防ぐ             |
| 「ついでに他の箇所も速くしておいた」         | 帰属不明な変更の束は、劣化時に切り戻せない。1 変更 1 計測を守る                  |
| 「ローカルで 1 回速くなったので完了」        | 1 回の計測はノイズと区別できない。複数回 + ばらつきの記録が最低条件              |
| 「テストは性能変更なので回さなくてよい」     | 並行化・キャッシュ・事前計算は正しさを壊す典型ルート。性能変更こそ全テストを回す |
| 「目標値が無いので速くなった気がすれば十分」 | 基準線と改善率の記録が無い報告は検証不能。現状値を測って基準線を作るのが先       |

## 6. 危険信号 (Red Flags)

- プロファイル結果を一度も見ずにコードを編集し始めている
- 前後比較の「前」の数値を取り忘れたまま変更を進めている
- ベンチマーク結果が実行のたびに大きく揺れるのに、都合の良い 1 回を報告に使おうとしている
- 「キャッシュを追加」した diff に無効化 (invalidation) の処理が無い
- 最適化 diff でテストコードの assert が緩められている

## 7. 完了チェックリスト (Verification)

- [ ] 変更前後の計測値 (コマンド・条件・複数回の結果) を summary に記録した
- [ ] ボトルネック特定に使ったプロファイル / 計測の証跡 (出力抜粋) がある。degrade した場合はその旨と限界を明記した
- [ ] 目標値との対比 (達成 / 未達 + 残ギャップ) を数値で書いた
- [ ] 既存テストが全て pass している (実行コマンドと結果を記録)
- [ ] 実装しなかった追加改善案・トレードオフを `[注意事項]` に残した
