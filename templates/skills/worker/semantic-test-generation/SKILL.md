# Semantic Test Generation

acceptance_criteria や要件からテストケースを体系的に導出するワークフロー。

## 1. テスト生成フロー

```
[要件分析] → [変数抽出] → [ケース分類] → [Given-When-Then化] → [実装]
                                                    → [評価→不足→追加ケース導出]
```

## 2. 要件からの変数抽出

acceptance_criteria を分解し、テスト対象の変数を特定する。

| ステップ | アクション | 出力 |
|---------|----------|------|
| S1 | criteria を文単位に分割 | 条件リスト |
| S2 | 各条件から入力・状態・出力を抽出 | 変数テーブル |
| S3 | 各変数の境界値・無効値を列挙 | 値域マップ |
| S4 | 変数間の依存関係を特定 | 組合せ制約 |

## 3. テストケース分類

| 種別 | 目的 | 優先度 | 導出元 |
|------|------|--------|--------|
| **Happy Path** | 正常系の基本動作 | 必須 | criteria の主条件 |
| **Boundary** | 境界値での動作 | 必須 | S3 の値域マップ |
| **Error Case** | 異常入力・障害時 | 必須 | S3 の無効値 |
| **Edge Case** | 稀な組合せ・競合 | 推奨 | S4 の組合せ制約 |

選択基準:
- 必須: 全タスクで最低限カバー
- 推奨: `persona_hint: quality-assurance` 時に追加

## 4. Given-When-Then テンプレート

各ケースを以下の形式で構造化する。

```
テスト名: [種別] 対象の動作説明
  Given: {前提条件・初期状態}
  When:  {実行するアクション}
  Then:  {検証する期待結果}
```

| 要素 | ルール |
|------|-------|
| Given | 再現に必要な最小限の前提のみ |
| When | 単一アクション（1テスト1操作） |
| Then | 検証可能な具体値・状態 |

## 5. 評価と補完

生成したテストセットを以下の観点で評価する。

| 観点 | 基準 | 不足時 |
|------|------|--------|
| **条件網羅** | criteria の全条件にテストあり | 未カバー条件からケース追加 |
| **種別バランス** | 3種別（正常/境界/異常）が各1以上 | 不足種別を補完 |
| **独立性** | 各テストが他に依存しない | 共有状態を排除 |

```
[テストセット] → [網羅チェック] → [全PASS] → [実装]
                               → [不足] → [追加導出→再チェック（最大2回）]
```

## 6. アンチパターン

| パターン | 問題 | 改善 |
|---------|------|------|
| **Happy Path のみ** | 異常系の見逃し | §3 の分類で最低3種別 |
| **巨大テスト** | 失敗原因の特定困難 | 1テスト1アクションに分割 |
| **実装依存** | リファクタで壊れる | 公開 API・振る舞いで検証 |
| **要件無視の網羅** | 不要ケースの大量生成 | criteria から逆算して導出 |
