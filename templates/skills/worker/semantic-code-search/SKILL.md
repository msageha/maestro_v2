# Semantic Code Search

コード検索を体系化し、影響範囲の見落としを防ぐための5段階分析手法。

## 1. 5段階分析フロー

```
[対象特定] → [パターン検索] → [参照分析] → [分類] → [検証]
```

| 段階 | 目的 | 主要ツール |
|------|------|-----------|
| **対象特定** | 変更対象のシンボル・型・パスを列挙 | Read, Glob |
| **パターン検索** | 直接参照・間接参照を網羅的に検索 | Grep, Glob |
| **参照分析** | 呼び出し元・依存先の関係を構造化 | Explore SubAgent |
| **分類** | 影響度で参照を分類 | Worker判断 |
| **検証** | 分類結果の正確性を確認 | Read |

## 2. トリガーフレーズ

以下のキーワードがタスクに含まれる場合、本スキルの適用を検討する:

| トリガー | 適用段階 |
|---------|---------|
| 「影響範囲」「どこで使われている」 | 全段階 |
| 「リファクタリング」「名前変更」 | 対象特定→参照分析 |
| 「テスト対象の特定」 | パターン検索→分類 |
| 「類似実装」「既存パターン」 | パターン検索 |

## 3. 検索パターン使い分け

| 検索目的 | 手法 | 例 |
|---------|------|-----|
| 関数・型の直接参照 | Grep（シンボル名） | `UserService` |
| ファイル構造の把握 | Glob（パスパターン） | `**/*_test.go` |
| 広範な依存関係調査 | Explore SubAgent | 複数パッケージ横断 |
| インターフェース実装 | Grep（メソッドシグネチャ） | `func.*Create\(` |

### ツール選択基準

```
検索対象
  ├─ 単一シンボル → Grep（直接実行）
  ├─ ファイル名パターン → Glob（直接実行）
  └─ 複数パッケージ横断 or 3回以上の検索 → Explore SubAgent
```

## 4. 影響度分類

参照を以下の3段階で分類し、Confidence Level を付与する:

| 影響度 | 定義 | Confidence | 対応 |
|--------|------|-----------|------|
| **High** | 型シグネチャ変更で直接コンパイルエラー | High | 必ず修正 |
| **Medium** | 動作は変わるがコンパイルは通る | Medium | テストで検証 |
| **Low** | 間接参照・コメント・ドキュメント | Low | 確認のみ |

## 5. 報告テンプレート

調査結果は以下の形式で `--summary` に構造化する:

```
[変更理由] {対象シンボル}の影響範囲を調査
[注意事項]
- High: {ファイル:関数} (直接参照 N箇所)
- Medium: {ファイル:関数} (間接参照 N箇所)
- Low: {ファイル} (コメント等 N箇所)
[未完了] なし
```
