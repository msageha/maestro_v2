# Frontend UI Implementation

Web UI (画面・コンポーネント・フォーム・レイアウト) を**実装・変更する**タスクで使う。「happy path が一応表示される」ではなく、**全状態・全入力デバイス・実レンダリング確認済み**を production 品質の完了条件とする。

## 1. 適用条件と棲み分け

| 本 skill が扱わないもの                      | 委ね先                                                             |
| -------------------------------------------- | ------------------------------------------------------------------ |
| 実装済み UI の E2E / ブラウザ検証の実施手順  | `webapp-testing` (本 skill は構築側。検証の報告様式はそちらに従う) |
| レンダリング性能・バンドルサイズの改善       | `performance-optimization`                                         |
| コンポーネント API (props/型契約) の設計判断 | `api-design` (公開コンポーネントの契約はそちらの規律で)            |
| TDD の進め方                                 | `tdd-red-green-refactor`                                           |

フレームワーク・デザインシステムは**プロジェクトの既存スタックに従う** (新規 UI ライブラリ・CSS フレームワークを無断導入しない)。ブラウザでの実確認手段 (dev server + Playwright / devtools) が使えない環境では、コンポーネントテスト・スナップショット・SSR 出力の確認まで degrade し、「実ブラウザ未確認」を `[注意事項]` に明記する。

## 2. 実装手順

1. **既存の部品と規約を先に探す**: 同種のコンポーネント・ページを 2〜3 個読み、デザイントークン (色・spacing・typography の変数)、命名、ディレクトリ構成、状態管理の流儀を抽出する。似た部品があるなら新作せず再利用・拡張する。
2. **状態を列挙してから作る**: 対象 UI の状態マトリクス (初期 / loading / 成功 (0 件・1 件・大量) / エラー / 権限なし / オフライン時など該当するもの) を先に書き出し、各状態の表示を決める。
3. **データ境界を型で固める**: API レスポンスの optional / null / 空配列を型どおりに扱う。「必ず値がある前提」の描画は最初の実データで壊れる。
4. **スタイルはトークン経由で**: 色・サイズの生値ハードコードではなく、既存のデザイントークン / テーマ変数を使う。無い値が必要なら追加を提案し `[注意事項]` に記す。
5. **実レンダリングで確認する**: dev server を起動し、主要 viewport (mobile / desktop) で各状態を実際に表示して確認する。確認方法と結果の報告は `webapp-testing` の様式に従う。

## 3. 全状態の実装 (happy path 禁止)

| 状態         | 最低要件                                                                      |
| ------------ | ----------------------------------------------------------------------------- |
| loading      | レイアウトシフトを起こさない placeholder / skeleton。無反応の空白にしない     |
| empty (0 件) | 「何もない」ことの明示と、次の行動 (作成ボタン等) への導線                    |
| error        | 利用者向けの言葉での通知と回復手段 (再試行・戻る)。生のエラー dump を出さない |
| 大量データ   | 溢れたときの挙動 (truncate / scroll / pagination) を決めて実装                |
| 極端な入力   | 長い文字列・多バイト文字・改行での折り返し崩れがない                          |

フォームを含む場合はさらに: 送信中の二重送信防止、フィールド単位のバリデーション表示、送信失敗時の入力値保持、を必須とする。

## 4. アクセシビリティ (後付けにしない)

- **セマンティック要素を優先する**: button は `<button>`、リンクは `<a>`、見出しは階層どおりの `<h*>`。`div` + onClick で代替しない。
- **すべての操作をキーボードで**: フォーカス可能・フォーカス順序が自然・フォーカスリングが見える。モーダルはフォーカストラップと Escape で閉じる。
- **ラベルと代替テキスト**: フォーム入力に label、アイコンのみのボタンに accessible name、意味のある画像に alt。
- **色だけに頼らない**: エラー・状態の伝達は色 + テキスト/アイコン。コントラストは既存トークンに従っていれば通る前提だが、生値を使った場合は確認する。
- 自動チェッカー (axe / Lighthouse a11y 等) が使えるなら 1 回かけ、使えなければ上記を手動確認して確認範囲を報告する。

## 5. よくある言い訳 (Common Rationalizations)

| 言い訳                                                    | なぜ間違いか                                                                                 |
| --------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| 「エラー状態はほとんど起きないので後で足す」              | 「後で」は来ない。エラー時に白画面になる UI は、最初の障害でそのまま利用者に露出する         |
| 「コードを見れば正しく描画されるはず。起動確認は省略」    | CSS 競合・null データ・viewport 依存の崩れはコードレビューでは見えない。表示して初めて分かる |
| 「アクセシビリティは専任のタスクでやるべき」              | 後付けの a11y 修正は構造の作り直しになる。セマンティクスとキーボード操作は実装時が最安       |
| 「デザインシステムに無い色だが、この 1 箇所だけ直書きで」 | 1 箇所の直書きが増殖して一貫性が崩れる。トークン追加の提案として上げるのが正しい経路         |
| 「デスクトップで確認したのでモバイルも大丈夫」            | レイアウト崩れの大半は狭い viewport で起きる。主要 viewport の確認は省略できない             |

## 6. 危険信号 (Red Flags)

- 既存コンポーネントを探さずに、似た部品をゼロから書き始めている
- `div` に onClick を付けてボタンの代わりにしようとしている
- API レスポンスの optional フィールドを non-null assertion や `!` で黙らせている
- 色・px 値の直書きが diff に並んでいる
- 一度もブラウザ (または degrade 先のコンポーネントテスト) で表示せずに completed と報告しようとしている

## 7. 完了チェックリスト (Verification)

- [ ] 状態マトリクス (loading / empty / error / 大量 / 極端入力のうち該当するもの) を列挙し、各状態の実装と確認結果を summary に記録した
- [ ] 実レンダリング確認を行った (viewport・確認手段・結果。degrade した場合は未確認範囲を明記)
- [ ] キーボードのみで主要フローを操作できることを確認した
- [ ] フォームがある場合、二重送信防止と失敗時の入力保持を確認した
- [ ] 新規の生値スタイル・新規依存を導入していない (導入が必要だった場合は `[注意事項]` で提案として報告した)
