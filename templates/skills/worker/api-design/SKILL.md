# API Design

公開 API (HTTP/RPC エンドポイント、ライブラリの公開関数、モジュール境界、CLI インターフェース、スキーマ) を**新設・変更する**タスクで使う。API は実装より寿命が長い。「動く」ではなく「あとから変えられる / 壊さず育てられる」を完了条件にする。

## 1. 適用条件と棲み分け

適用するのは、他者 (別モジュール・別チーム・外部利用者) が呼ぶ境界の設計・変更を含むタスク。内部関数のリファクタだけなら適用外。

| 本 skill が扱わないもの                      | 委ね先                                                              |
| -------------------------------------------- | ------------------------------------------------------------------- |
| 変更の参照元列挙・循環依存チェックの実施手順 | `code-structure-discipline` (本 skill は「何を契約とするか」の設計) |
| JSON/YAML 出力インスタンスのスキーマ検証     | `structured-output-lifecycle`                                       |
| 既存 API の仕様調査・出典付き報告            | `source-grounded-response`                                          |
| 廃止・移行の複数フェーズ計画立案             | Planner の migration 計画 (本 skill は 1 タスク内の互換判断)        |

スキーマ定義は既存の形式 (OpenAPI / protobuf / JSON Schema / 型定義) をまず探して従う。形式化された仕様が無いプロジェクトでは、**型シグネチャ + docstring/コメントでの契約明文化**まで degrade し、仕様書が存在しない旨を `[注意事項]` に書く。

## 2. 設計手順

1. **利用側から書き始める**: 実装の都合ではなく、呼び出しコード例を先に書く (README / テスト / docstring の形で)。呼びにくい API はこの時点で露見する。
2. **契約を明文化する**: 各操作について「入力の制約 / 出力の保証 / エラーの種類 / 冪等性 / 副作用」を仕様として書く。暗黙の前提 (順序保証、null の扱い、タイムゾーン、単位) を残さない。
3. **既存の語彙・規約に揃える**: 命名 (リソース名、動詞、大文字小文字)、ページネーション方式、エラー形式、認証の渡し方は既存 API から抽出して従う。1 つの API だけ流儀が違うのは設計バグ。
4. **エラー契約を先に決める**: 利用側が分岐できる形 (機械可読なエラーコード / 型) で分類する。「500 + メッセージ文字列」だけの契約は、利用側にメッセージ文字列の parse を強いる。
5. **最小の表面積で出す**: 「あとで使うかもしれない」フィールド・オプション・エクスポートは出さない。公開したものは取り消せないが、追加はいつでもできる。
6. **契約をテストに固定する**: 公開境界の入出力・エラー分類をテストで固定する。契約テストが無い API は、次の変更者が何を守るべきか分からない。

## 3. 互換性の判定基準

変更タスクでは、diff を以下で分類してから実装する:

| 変更                                         | 判定                                              |
| -------------------------------------------- | ------------------------------------------------- |
| 省略可能なフィールド・パラメータの追加       | 互換 (デフォルト値の意味を明記する)               |
| レスポンスへのフィールド追加                 | 概ね互換 (利用側の strict decode を確認)          |
| フィールドの削除・改名・型変更・意味変更     | **破壊的**                                        |
| 必須パラメータの追加、バリデーションの厳格化 | **破壊的** (今まで通っていた入力が落ちる)         |
| エラーコード・ステータスの変更、例外の型変更 | **破壊的** (利用側の分岐が壊れる)                 |
| 挙動の順序・タイミング・デフォルト値の変更   | **破壊的になり得る** (暗黙依存を調査してから判定) |

破壊的変更が必要な場合、無断で実施しない。選択肢 (バージョン分離 / 新旧並存 + 非推奨化 / 破壊的変更の断行) とその影響を `[注意事項]` に提示し、タスクが明示的に許可している場合のみ実施する。許可済みの場合も、旧契約の利用箇所を列挙して追随修正するか、移行手順を成果物に含める。

## 4. アンチパターン (禁止)

- **実装 leak**: 内部の DB カラム名・内部例外・ORM オブジェクトをそのまま境界に露出する
- **boolean 引数の増殖**: `f(true, false, true)`。意味のある型・オプション構造体・別関数に分ける
- **何でも受ける入力**: `map[string]any` / 無検証の dict を公開境界の入力型にする
- **部分的成功の曖昧な返却**: 一括操作でどれが成功しどれが失敗したか判別できない戻り値
- **サイレント仕様変更**: 契約文書・スキーマを更新せずに挙動だけ変える

## 5. よくある言い訳 (Common Rationalizations)

| 言い訳                                             | なぜ間違いか                                                                                   |
| -------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| 「利用者はまだいないので互換性は考えなくてよい」   | 公開した瞬間から利用は始まる。最初の形が事実上の v1 契約になり、直すコストは今が最小           |
| 「フィールドを増やしておけば将来困らない」         | 使われないフィールドも契約になり、削除は破壊的変更になる。表面積は負債である                   |
| 「エラーメッセージを読めば分かるので分類は不要」   | メッセージ文字列は契約ではなく変わり得る。利用側の文字列マッチ依存を生んだ時点で改善不能になる |
| 「内部モジュール間だから契約の明文化は大げさ」     | モジュール境界は将来のチーム境界・サービス境界。暗黙契約は境界を跨いだ瞬間に事故になる         |
| 「破壊的だが利用箇所は少ないので黙って変えてよい」 | 少数でも壊れるものは壊れる。判定と選択肢の提示は Worker の責務、断行の判断は Planner の責務    |

## 6. 危険信号 (Red Flags)

- 呼び出し例を 1 つも書かないまま、実装の内部構造から API の形を決めている
- 既存 API の命名・エラー形式を確認せずに、新しい流儀を持ち込んでいる
- 公開シグネチャの変更 diff に、契約テスト・スキーマ・ドキュメントの更新が伴っていない
- 「互換かどうか分からないがたぶん大丈夫」で変更を進めている
- 公開境界の入力検証を「呼ぶ側が正しく渡す前提」で省略している

## 7. 完了チェックリスト (Verification)

- [ ] 契約 (入力制約 / 出力保証 / エラー分類 / 冪等性) が成果物 (スキーマ・型・docstring のいずれか) に明文化されている
- [ ] 利用側視点の呼び出し例が存在し、実際に動くことを確認した
- [ ] 変更タスクでは互換性判定 (互換 / 破壊的) と、その根拠・対応を summary に記録した
- [ ] 公開境界の契約テストが pass している (実行コマンドと結果を記録)
- [ ] 既存 API 規約と揃えたこと、または規約不在で流儀を新設した旨を `[注意事項]` に明記した
