# Structured Output Lifecycle

構造化出力（JSON、YAML、summary タグ等）の生成→検証→修復ループを体系化する。

## 1. 出力生成フロー

```
[要件確認] → [スキーマ定義] → [出力生成] → [検証] → [PASS] → [送出]
                                              → [FAIL] → [修復→再検証（最大3回）]
                                                  → [修復不能→failed報告]
```

## 2. スキーマ定義パターン

出力前に期待構造を明確化する。

| 対象 | スキーマ例 | 検証ポイント |
|------|----------|------------|
| **summary 報告** | `[変更理由]` `[注意事項]` `[未完了]` の3タグ | タグの存在・順序・内容非空 |
| **JSON 出力** | 必須キー・型・値域 | キー存在・型一致・null 許容 |
| **CLI 引数** | フラグ名・値形式 | 必須フラグ・値のエスケープ |
| **コード生成** | 構文・import・型整合 | ビルド通過・lint エラーなし |

## 3. 検証レベル

段階的に検証し、早期に問題を検出する。

| Level | 検証内容 | 手法 | 例 |
|-------|---------|------|-----|
| L1 **構造** | 必須フィールド存在 | パターンマッチ | タグ・キーの存在確認 |
| L2 **型** | 値の型・形式 | 型チェック | 文字列/数値/配列の一致 |
| L3 **意味** | 値の妥当性 | ドメインルール | 範囲内・参照先存在 |
| L4 **整合** | 他出力との一貫性 | クロスチェック | files-changed と実変更の一致 |

## 4. 自動修復パターン

| 障害 | Severity | 修復アクション |
|------|----------|-------------|
| タグ欠損 | Critical | テンプレートから補完 |
| 型不一致 | High | 型変換・キャスト |
| 値域外 | Medium | クランプ or デフォルト値 |
| フォーマット崩れ | Low | 再整形 |

修復ルール:
- **Critical/High**: 自動修復→再検証→失敗時は手動修正
- **Medium/Low**: 自動修復→再検証→ログ記録

## 5. Worker 適用例

### summary 報告時

```
1. テンプレート準拠で summary 生成
2. L1: 3タグ（変更理由/注意事項/未完了）の存在確認
3. L2: 各タグの内容が空でないか確認
4. L3: files-changed と実際の変更ファイルが一致するか確認
5. FAIL → タグ補完・内容修正 → 再検証
```

### コード生成結果

```
1. 生成コードの構文検証（ビルド/lint）
2. import 不足 → 自動追加
3. 型エラー → 型定義確認・修正
4. 2回失敗 → アプローチ変更
```

## 6. アンチパターン

| パターン | 問題 | 改善 |
|---------|------|------|
| **検証スキップ** | 構造破損に気づかない | 全出力に L1 検証を必須化 |
| **無限修復** | 修復→失敗ループ | 最大3回で打ち切り |
| **過剰検証** | 全フィールドに L4 | リスクに応じてレベル選択 |
