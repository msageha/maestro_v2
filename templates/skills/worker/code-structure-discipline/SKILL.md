# Code Structure Discipline

ファイルを「読んでから考える」のではなく、**構造 (シンボル・参照・依存の向き) を先に照会してから最小限を読む**規律。変更の前後に構造の観点の点検を挟むことで、token を節約しつつ、参照元の見落としと構造劣化 (循環依存・dead code) を防ぐ。

## 1. 適用条件と棲み分け

**適用するのは、複数ファイル・モジュール境界を跨ぐ非自明な変更のみ。** 1-2 ファイルで完結する自明な修正・文言変更にこの規律を課さない (過剰分析は token と時間の浪費)。

| 本 skill が扱わないもの      | 委ね先                                                              |
| ---------------------------- | ------------------------------------------------------------------- |
| 検索の手法・影響度分類・報告 | `semantic-code-search` (5 段階分析フロー)。本 skill は再実装しない  |
| タスク依存 DAG の循環検出    | daemon の plan 検証 (コード構造の循環とは別レイヤ)                  |
| 一回性の全体監査レポート     | `tech-debt-audit` (本 skill は変更ごとの規律であり、監査は別タスク) |

接地はランタイムの **LSP ツール** (定義参照・参照列挙・シンボル検索) を使う。専有の code-graph API・外部グラフサービスは導入しない。LSP が使えない環境 (未対応言語・サーバ不在) では Grep / Explore SubAgent に degrade し、**構造照会を grep で代替した旨を summary に明示する** (参照の網羅性が落ちるため)。

## 2. 規律 1: 読む前に照会する (token 節約)

ファイル全文の Read を積む前に、LSP で構造を掴む:

1. 変更対象シンボルの**定義位置**を LSP で特定する (ファイル名の推測で Read しない)
2. **参照一覧・呼出関係**を LSP で取得し、変更に関係する箇所だけを絞り込む
3. 絞り込んだ箇所のみ Read する。「とりあえず全文読む」は最後の手段

構造照会 1 回は全文 Read 数回分の token を節約する。`context-window-efficiency` の延長として、本体コンテキストの温存を優先する。

## 3. 規律 2: 編集前に impact を列挙する

シグネチャ・型・公開インターフェース・振る舞いを変える前に:

1. 変更対象の**参照元を LSP で列挙**し、影響を受ける caller / dependent を特定する
2. 影響箇所ごとの対応 (追随修正 / 影響なしの根拠 / スコープ外として報告) を実装計画に含める
3. 列挙した impact を `--summary` の `[注意事項]` に記録する (後続タスク・Planner の判断材料になる)

「参照元は多分ここだけ」という推測で編集を始めない。列挙してから編集する。

## 4. 規律 3: commit・報告前に構造を検証する

変更によって**構造が劣化していない**ことを、完了報告の前に確認する:

- **循環依存**: 新しく追加した import / 依存が、逆向き参照 (下位層→上位層) や循環を作っていないか。言語ネイティブの検出 (コンパイラの import cycle エラー・lint) を先に使い、無ければ依存の向きを LSP / grep で確認する
- **dead code**: 置換・削除したシンボルの旧定義や参照残骸が残っていないか。旧シンボル名で LSP 参照 / grep し、ヒット 0 を確認する
- **層の整合**: 変更がプロジェクトの層構造 (例: domain 層が infra 層を参照しない) を破っていないか。既存の依存の向きに従う

違反を見つけたら修正してから報告する。修正がスコープ外に及ぶ場合は、違反の内容を `[注意事項]` に記載して判断を Planner に委ねる。

## 5. よくある言い訳 (Common Rationalizations)

| 言い訳                                                 | なぜ間違いか                                                                                         |
| ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------- |
| 「ファイルを読めば構造も分かるので照会は二度手間」     | 全文 Read は構造の一部しか示さず token を大量消費する。照会→絞込→Read の順が結局最短                 |
| 「参照元はコンパイルエラーで分かるので事前列挙は不要」 | コンパイルが検出するのは型不整合だけ。振る舞い変更・動的参照・別ビルドターゲットは沈黙したまま壊れる |
| 「循環依存はビルドが通ったので存在しない」             | 言語によっては循環がビルドエラーにならない (実行時解決・遅延 import)。依存の向きは明示的に確認する   |
| 「小さな変更なので構造検証を省略する」                 | 1-2 ファイルの自明変更なら本 skill 自体が適用外。適用対象と判断した変更で検証だけ省くのは矛盾        |

## 6. 危険信号 (Red Flags)

- 変更対象の定義位置を確認する前に、ディレクトリ内のファイルを推測で次々に Read している
- シグネチャを変えた後になってから参照元を探し始めている
- import を追加するとき、依存の向きを考えずに「動くから」で通そうとしている
- LSP が使えないのに、その旨を報告せず grep 結果を「全参照」として扱っている

## 7. 完了チェックリスト (Verification)

- [ ] 変更対象の参照元を編集**前**に列挙し、各参照の扱いを決めた (記録が `[注意事項]` にある)
- [ ] 旧シンボル・置換前の参照の残骸が無い (LSP 参照 / grep でヒット 0 を確認した)
- [ ] 新規に追加した依存が循環・逆向き参照を作っていない (確認手段を summary に記録)
- [ ] LSP を使えなかった場合、grep degrade した旨と網羅性の限界を summary に明示した
