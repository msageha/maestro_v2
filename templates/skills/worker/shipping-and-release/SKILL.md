# Shipping and Release

機能・サービスの**リリース工程の設計・準備・実施**を扱うタスクで使う (リリース手順書、ロールアウト計画、feature flag 導入、rollback 整備など)。原則: **rollback 手順の無いリリース計画は未完成**であり、「出す手順」と「戻す手順」は常に対で作る。

## 1. 適用条件と棲み分け

| 本 skill が扱わないもの                  | 委ね先                                                           |
| ---------------------------------------- | ---------------------------------------------------------------- |
| CHANGELOG・リリースノートの文書作成      | `docs-changelog-release-notes` (本 skill は工程・判断基準の設計) |
| リリースを実行するパイプライン自体の実装 | `ci-cd-pipelines`                                                |
| リリース後の監視指標・アラートの実装     | `observability` (本 skill はどの指標を見て判断するかの選定)      |
| 複数リリースに跨る廃止・移行の長期計画   | Planner の migration 計画                                        |

破壊的操作の境界に注意する: **実環境へのデプロイ・publish・タグ push はタスクが明示的に指示した場合のみ実行する**。指示が無い場合の成果物は「実行可能な手順・スクリプト・チェックリスト」であり、Worker が自発的に本番操作を行わない。段階的ロールアウト基盤 (feature flag サービス、canary 機構) が無いプロジェクトでは、環境分離 (staging → 本番) や設定値によるオン/オフまで degrade し、粒度の限界を `[注意事項]` に明記する。

## 2. Pre-launch checklist の作り方

リリース準備タスクでは、以下の観点を具体的な確認項目 (誰でも実行できるコマンド・確認先つき) に落とす:

1. **検証**: テスト・quality gate が対象コミットで green であること。「最新 main で green」ではなく「リリースするコミットで green」を確認する。
2. **互換性**: DB マイグレーション・API 変更・設定変更の互換性。特にマイグレーションは「コードより先に適用して安全か / 戻すときどうするか」を明記する。
3. **依存関係**: 先にリリースが必要な依存 (他サービス、インフラ変更、シークレット登録) の有無と順序。
4. **観測**: リリース後に健全性を判断する指標 (エラー率・レイテンシ・業務指標) と、その確認方法 (ダッシュボード URL・コマンド)。
5. **rollback**: 戻し方 (後述)。
6. **告知**: 影響を受ける利用者・チームへの通知要否。

チェックリストは「□ テストが通っている」のような曖昧な項目にせず、「□ `<コマンド>` が exit 0 (実行者: リリース担当)」のように検証可能に書く。

## 3. 段階的ロールアウトの設計

- **爆発半径を段階で制御する**: internal → canary (数%) → 段階拡大 → 全量、を基本形とし、プロジェクトで使える粒度 (flag / % / 環境 / テナント) に合わせる。
- **各段階に「進む条件」と「止める条件」を数値で決める**: 「canary のエラー率が基準線 + X% 未満を Y 分維持したら次へ」「超えたら rollback」。条件の無い段階的リリースは、ただの遅いリリースである。
- **変更は 1 リリース 1 関心事に**: 大きな変更ほど、リリース単位を分割して個別に rollback 可能にする。データマイグレーションとコード変更は原則別リリースにする (expand → migrate → contract)。
- **feature flag は導入時に削除条件を決める**: flag 名、デフォルト値、全量到達後の削除タイミングを記録する。消されない flag は分岐の負債になる。

## 4. Rollback 戦略

1. **手段を明示する**: revert deploy / flag off / 設定切り戻し、のどれで戻すか。所要時間の目安も書く。
2. **戻せないものを列挙する**: 適用済み DB マイグレーション、送信済み通知、外部への書き込み。戻せない操作を含むリリースは、前方修正 (fix-forward) の手順も用意する。
3. **rollback の発動条件を事前に決める**: 「どの指標がどうなったら戻すか」をリリース前に固定する。障害中にその場で判断基準を作らない。
4. **可能なら rollback を予行する**: staging での切り戻し実行、あるいは最低限 dry-run。「書いてあるが試していない手順」はリスクとして明記する。

## 5. よくある言い訳 (Common Rationalizations)

| 言い訳                                                 | なぜ間違いか                                                                                    |
| ------------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| 「小さい変更なので rollback 手順は不要」               | 小さい変更の事故は「準備が無い」ぶん復旧が遅い。手段が revert 一発なら、そう書くだけで済む      |
| 「問題が起きたらその場で判断すればよい」               | 障害中は判断力が最も落ちている。進む/止まる条件を事前に数値で決めるのが段階的ロールアウトの本体 |
| 「みんな急いでいるので canary を飛ばして全量で出す」   | 全量事故のコストは canary の待ち時間の比ではない。急ぐなら段階の時間幅を縮める方が正しい        |
| 「flag はいつか消すので今は記録しなくてよい」          | 削除条件を決めなかった flag は消えない。導入した本人が削除条件を書くのが最安                    |
| 「マイグレーションは戻せないが、まあ失敗しないだろう」 | 戻せない操作こそ expand/contract 分割・バックアップ・fix-forward 手順の対象。楽観は戦略ではない |

## 6. 危険信号 (Red Flags)

- リリース手順書に rollback セクションが無い、または「revert する」の一行しかない
- 段階的ロールアウトの「次に進む条件」が誰の判断かも数値も決まっていない
- コード変更と非互換マイグレーションが同一リリースに同居している
- タスクで指示されていないのに、本番環境への deploy・publish・タグ push を実行しようとしている
- チェックリストの項目が「〜を確認する」ばかりで、確認コマンド・確認先が書かれていない

## 7. 完了チェックリスト (Verification)

- [ ] 成果物に pre-launch checklist (検証可能な項目 + 実行コマンド/確認先) が含まれている
- [ ] ロールアウトの各段階に「進む条件 / 止める条件」が数値で定義されている (基盤が無い場合は degrade 内容を明記)
- [ ] rollback 手順 (手段・所要時間・戻せない操作の列挙) が対で存在する
- [ ] rollback 手順の予行または dry-run の結果、未予行ならそのリスクを summary に記録した
- [ ] feature flag を導入した場合、flag 名・デフォルト値・削除条件を記録した
- [ ] 実環境への破壊的操作は、タスクの明示指示があった場合のみ実行した (実行有無を summary に明記)
