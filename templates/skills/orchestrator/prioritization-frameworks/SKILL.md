# Prioritization Frameworks

Orchestratorが複数の要求を受けた際に、Plannerへの委譲順序を判断するためのフレームワーク集。ツール実行不要で、判断ロジックのみで適用可能なものを選定している。

## 1. ICE スコアリング

各要求に Impact / Confidence / Ease を 1-5 で評価し、積でスコア化する。

| 軸 | 評価基準 | 5（高） | 1（低） |
|----|---------|---------|---------|
| **Impact** | 要求実現時の効果 | コア機能に直結 | 補助的な改善 |
| **Confidence** | 要求の明確さ | 具体的で曖昧さなし | 追加確認が必要 |
| **Ease** | 実装の容易さ | 1タスクで完結 | 複数フェーズ必要 |

```
スコア = Impact × Confidence × Ease
高スコア順に委譲する
```

**適用場面:** 性質の異なる要求が3件以上ある場合

## 2. MoSCoW 分類

要求を4カテゴリに分類し、Must → Should → Could の順で委譲する。

| カテゴリ | 判断基準 | Orchestratorの行動 |
|---------|---------|-------------------|
| **Must** | ユーザーが明示的に必須と指定 / エラー修正 | 即座にPlannerへ委譲 |
| **Should** | 重要だが代替手段がある | Must完了後に委譲 |
| **Could** | あれば良いが影響は限定的 | リソースに余裕があれば委譲 |
| **Won't** | 今回スコープ外とユーザーが同意 | 委譲しない。記録のみ |

**適用場面:** ユーザーの要求にスコープの優先度差がある場合

## 3. Eisenhower マトリクス

緊急度と重要度の2軸で分類し、象限ごとに対応を決める。

|  | 重要 | 重要でない |
|--|------|-----------|
| **緊急** | 即時委譲（エラー修正、ブロッカー解消） | ユーザーに確認後判断 |
| **緊急でない** | 計画的に委譲（設計変更、リファクタリング） | 後回し or スコープ外 |

**適用場面:** 割り込み要求と既存作業の優先度を比較する場合

## 4. Maestro固有の判断基準

上記フレームワークに加え、以下のMaestro固有基準を考慮する。

### 依存関係

| 状態 | 判断 |
|------|------|
| 他コマンドの結果を待つ必要がある | 待機。先行コマンド完了後に委譲 |
| 独立して実行可能 | 即座に委譲可能 |
| 先行タスクの成果物を入力とする | 先行完了を確認してから委譲 |

### 並列可能性

| 状態 | 判断 |
|------|------|
| 変更対象ファイルが重複しない | 並列コマンドとしてPlannerに委譲 |
| 変更対象が重複する可能性あり | 逐次実行。先行完了後に委譲 |
| 読み取り専用の調査タスク | 他コマンドと並列可能 |

### リスク評価

| リスクレベル | 条件 | 対応 |
|------------|------|------|
| **高** | 破壊的変更、DB操作、外部API呼び出し | ユーザー確認を取ってから委譲 |
| **中** | 複数ファイル変更、設定変更 | 通常委譲。結果を詳細報告 |
| **低** | 単一ファイル変更、テスト追加 | 即座に委譲 |

## 5. 判断フロー

1. MoSCoWで分類 → 2. 依存関係確認 → 3. 高リスクはユーザー確認 → 4. Must内はICE/Eisenhowerで順序付け → 5. 独立要求は並列委譲 → 6. 実行

**原則:** 迷ったらユーザーに確認。判断根拠は報告に含める。フレームワークは補助であり最終判断はユーザーの意図に従う。
