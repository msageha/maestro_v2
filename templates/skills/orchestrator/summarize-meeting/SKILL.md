# Summarize Meeting — コマンド完了結果の構造化要約

コマンド実行完了後に、dashboard.md と results/*.yaml の情報を構造化要約として整理する。ユーザーへの報告品質を向上させ、次のアクションを明確にする。

## 入力ソース

| ソース | 取得情報 |
|--------|---------|
| `dashboard.md` | コマンド全体の進捗・ステータス |
| `results/*.yaml` | 各タスクの summary, files_changed, learnings |

## 構造化要約テンプレート

コマンド完了時に以下の構造で要約を作成する。

### 1. コマンド概要

```
- 元の要求: {ユーザーが入力した元のコマンド/要求}
- 実行日時: {開始〜完了の時間帯}
- 全体ステータス: {completed / partial / failed}
```

### 2. 成果サマリ

| タスク | ステータス | 主要成果物 |
|--------|-----------|-----------|
| {task_purpose} | {completed/failed} | {files_changed の要約} |

- 完了タスク数・失敗タスク数を明記する
- files_changed が多い場合はカテゴリ別に集約する

### 3. 決定事項

タスク実行中に Worker/Planner が行った設計判断・選択を列挙する。

```
- {判断内容}: {選択した理由}（タスク: {task_id}）
```

results の summary 内 `[変更理由]` タグから抽出する。

### 4. 未解決事項

```
- {未解決の内容}（原因: {失敗理由 or 部分完了の詳細}）
```

以下を対象とする:
- `--status failed` のタスク
- `--partial-changes` 付きのタスク
- summary 内 `[未完了]` タグの内容

### 5. 次のアクション

```
- [ ] {推奨する後続コマンドや操作}
- [ ] {ユーザー判断が必要な事項}
```

## 適用タイミング

- 全タスクが完了/失敗した後の最終報告時に使用する
- 部分的完了の中間報告にも適用可能
- ユーザーが「結果を教えて」「何が変わった？」と聞いた際にも活用する

## 品質基準

| 基準 | 内容 |
|------|------|
| 網羅性 | 全タスクの結果が含まれていること |
| 簡潔性 | 各セクション 3-5 行以内を目安とする |
| 行動可能性 | 「次のアクション」が具体的であること |
| 追跡可能性 | 各項目が元の task_id/結果に紐付くこと |
