# PRD — 要件の初期構造化スキル

ユーザーの曖昧な要求を対話ベースで明確化し、Planner に委譲可能な構造化コマンドを生成する。技術的な分解は行わず「何を作るか」の整理に特化する。

> **create-prd（Planner スキル）との差別化**: Orchestrator は要件の初期構造化のみ担当する。技術スタック選定・タスク分解・依存関係設計は Planner に委譲する。

## Phase 1: Discovery（ユーザーへの質問）

以下の 5 軸でユーザーに質問し、要件を明確化する。一度に全て聞かず、回答に応じて深掘りする。

### 1. 問題の明確化

```
- 何を解決したいのか？（現状の課題）
- 誰のための機能か？（ユーザー/システム/開発者）
- なぜ今必要か？（背景・動機）
```

### 2. 成功基準

```
- どうなれば完了か？（具体的な完了条件）
- 成功をどう測定するか？（動作確認方法）
- 既存の類似実装はあるか？（参考になるもの）
```

### 3. 制約

```
- 技術スタックの制約はあるか？
- 期限・リソースの制約はあるか？
- 影響範囲の制限はあるか？（変更してよい範囲）
```

### 4. スコープ

```
- やること（In Scope）: {具体的な機能・変更}
- やらないこと（Out of Scope）: {明示的に除外する範囲}
- 将来的にやるかもしれないこと（Future）: {今回は対象外だが意識する}
```

### 5. 優先度

| 優先度 | 定義 | 項目 |
|--------|------|------|
| **Must** | これがないと意味がない | {必須機能} |
| **Should** | 強く期待するが代替可能 | {推奨機能} |
| **Could** | あれば嬉しいが省略可能 | {任意機能} |

## Phase 2: Planner 委譲コマンドの構造化

Discovery で得た情報を以下のテンプレートで構造化し、Planner へのコマンドとして出力する。

```
## 要求概要
{1-2文で要求を要約}

## 背景・動機
{なぜ必要か}

## 成功基準
- {条件1}
- {条件2}

## スコープ
- In: {やること}
- Out: {やらないこと}

## 制約
- {制約1}
- {制約2}

## 優先度
- Must: {必須項目}
- Should: {推奨項目}
- Could: {任意項目}
```

## 適用タイミング

- ユーザーが「〜を作りたい」「〜を改善したい」と要求した際
- 要求が曖昧で、そのまま Planner に委譲すると手戻りが発生しそうな場合
- ユーザーが「PRD を作って」「要件を整理して」と明示的に依頼した場合

## 品質基準

| 基準 | 内容 |
|------|------|
| 明確性 | 各項目が曖昧語（「適切に」「うまく」）を含まないこと |
| 完全性 | 5 軸すべてに回答があること（不明は「未定」と明記） |
| 行動可能性 | 構造化コマンドがそのまま Planner に渡せること |
| 簡潔性 | 全体で A4 1 ページ以内を目安とする |
