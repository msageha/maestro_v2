# Internal Communications

Orchestratorがユーザーへ報告する際の構造化テンプレート集。dashboard.md や results/*.yaml から読み取った情報を、簡潔かつ網羅的に伝える。

## 1. コマンド完了報告

コマンドの全タスクが完了した際に使用する。

```
## 完了: {command_purpose}

| 項目 | 内容 |
|------|------|
| コマンドID | {command_id} |
| タスク数 | {total}件（成功: {ok} / 失敗: {fail}） |
| 主な成果物 | {変更ファイル・生成物の要約} |

### 変更ファイル一覧
- {file1}: {変更内容}
- {file2}: {変更内容}

### 未解決事項
- {あれば記載。なければ「なし」}
```

**記載ルール:**
- 成果物は具体的パスで列挙する
- 未解決事項は後続アクションの提案を含める
- 失敗タスクがある場合はエラー報告セクションも併記する

## 2. 進捗中間報告

長時間コマンドの途中経過をユーザーに伝える際に使用する。

```
## 進捗: {command_purpose}

| 状態 | 件数 |
|------|------|
| 完了 | {completed} |
| 実行中 | {running} |
| 待機中 | {pending} |

### 現在の作業
- {worker_id}: {task_purpose}

### ブロッカー
- {あれば記載。なければ「なし」}
```

**記載ルール:**
- ブロッカーがある場合はユーザー判断が必要かを明記する
- 推定残時間は記載しない（不正確になるため）

## 3. エラー/失敗報告

タスク失敗やリトライ不能な問題が発生した際に使用する。

```
## 失敗: {command_purpose}

| 項目 | 内容 |
|------|------|
| 失敗タスク | {task_id}: {task_purpose} |
| 原因要約 | {error_summary} |
| 部分変更 | {あり/なし} |

### 推奨アクション
1. {具体的な対処案}
2. {代替案があれば}
```

**記載ルール:**
- 原因は技術詳細を含めつつ1-2文に収める
- 部分変更ありの場合、変更済みファイルを列挙する
- 推奨アクションは実行可能な具体案を提示する

## 4. 継続モード報告

前回の結果を踏まえて次のアクションを提案する際に使用する。

```
## 継続提案: {context}

### 前回結果サマリ
- {完了した内容の要約}

### 次アクション提案
1. {提案1}: {理由}
2. {提案2}: {理由}

### ユーザー判断が必要な事項
- {判断事項と選択肢}
```

**記載ルール:**
- 前回結果は要点のみ（詳細は完了報告を参照）
- 提案は優先度順に並べ、理由を付記する
- ユーザー判断事項は選択肢を明示して意思決定を容易にする

## 5. 情報構造化パターン

dashboard.md / results/*.yaml から報告を組み立てる際の指針。

| ソース | 抽出情報 | 報告での用途 |
|--------|---------|-------------|
| dashboard.md | 全体進捗・フェーズ状態 | 進捗報告の件数 |
| results/*.yaml | status, summary, files_changed | 完了/失敗報告の詳細 |
| results/*.yaml | learnings | 継続提案の根拠 |

**原則:**
- 生データをそのまま貼り付けない。要約して構造化する
- ユーザーが次のアクションを判断できる情報を優先する
- 技術詳細は必要最小限に留め、求められたら追加する
