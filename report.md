# Maestro v2 コードベース品質レビューレポート

## 1. エグゼクティブサマリ

Maestro v2 のコードベース全体を対象に、4つの調査チームによる並行レビューを実施した。

### 全体評価

コードベースは全体的に堅牢な設計がなされているが、Worktree/Git 操作周りに **Critical 1件、High 2件** の実質的なバグが存在する。これらは本番運用時にデータ損失やリソースリークを引き起こす可能性があり、優先的な対応が必要である。

Medium 以下の問題は設計上のトレードオフや将来的な改善候補であり、即時対応の緊急性は低い。

| 重要度 | 件数 |

## 3. 発見事項一覧

### Critical

#### C-1: PublishToBase が未コミット変更を無警告で破棄

- **ファイル**: `manager.go:809`
- **問題**: `PublishToBase` が base branch を checkout する際に working tree を強制リセットし、未コミットの変更を破棄する。Worker が作業中のファイルが失われる可能性がある。
- **Codex 見解**: 実バグ。`git status --porcelain` で事前チェックし、未コミット変更がある場合はエラーを返すべき。
- **推奨対応**: checkout 前に `git status --porcelain` で dirty check を実施し、未コミット変更がある場合は操作を中断してエラーを返す。

---

### High

#### H-1: クリーンアップ時の branch -D エラー無視による orphan ブランチ残存

- **ファイル**: `manager.go:869,1369`
- **問題**: worktree クリーンアップ時に `git branch -D` のエラーを無視するため、削除に失敗したブランチが永久に残存する。長期運用でブランチが蓄積し、リソースを圧迫する。
- **Codex 見解**: ログ出力とステート管理の改善を推奨。
- **推奨対応**: エラー発生時にログを出力し、ステートファイルに未削除ブランチを記録する。定期的なリコンシリエーションで再削除を試みる仕組みを追加する。

#### H-2: GC の CreatedAt パース失敗によるサイレントスキップ

- **ファイル**: `manager.go:936-938`
- **問題**: GC（ガベージコレクション）で `CreatedAt` のパースに失敗すると、その worktree をサイレントにスキップする。TTL と max 両方の GC ルールから漏れ、worktree が永久に残存する。
- **Codex 見解**: ログ出力と `mtime` フォールバックを推奨。
- **推奨対応**: パース失敗時に警告ログを出力し、ファイルシステムの `mtime` をフォールバック値として使用する。

---

### Medium

#### M-1: 結果処理ループ上限 100 の制限

- **ファイル**: `result_handler.go:216,311`
- **問題**: 結果処理ループに固定上限 100 が設定されており、大量の結果が同時に発生した場合に処理が遅延する可能性がある。
- **Codex 見解**: 時間ベースのドレインループへの移行を推奨。
- **推奨対応**: 固定上限の代わりに時間ベース（例: 100ms 以内で処理可能な分だけドレイン）のループに変更する。

#### M-2: Heartbeat Lease の TOCTOU 競合

- **ファイル**: `dispatcher.go`, `task_heartbeat_handler.go:94`
- **問題**: Heartbeat 処理で lease のチェックと更新の間に TOCTOU（Time-of-Check-to-Time-of-Use）競合が存在する。
- **Codex 見解**: 実在する懸念。`scanMu.RLock` のブロッキング取得を推奨。
- **推奨対応**: lease チェックと更新をアトミックに行うか、`scanMu.RLock` でブロッキング取得に変更する。

#### M-3: Priority フィールドの Integer Overflow

- **ファイル**: `dispatcher.go:203`
- **問題**: Priority 値に対する境界チェックがなく、極端な値が入力された場合に integer overflow が発生する可能性がある。
- **Codex 見解**: `int64` クランプを推奨。
- **推奨対応**: Priority 値を受け取る際に `int64` の範囲でクランプ処理を追加する。

---

### Low

#### L-1: recoverPanic 内の go d.Shutdown()

- **ファイル**: `daemon_helpers.go:20`
- **問題**: panic リカバリ内で goroutine として `Shutdown()` を呼び出しているが、エラーハンドリングが不十分。
- **Codex 見解**: 現状問題なし。
- **推奨対応**: 現状維持。必要に応じてログ出力を追加。

#### L-2: シグナルリスナー goroutine リーク

- **ファイル**: `daemon_shutdown.go:31-47`
- **問題**: シグナルリスナーの goroutine がシャットダウン後もリークする可能性がある。
- **Codex 見解**: オプショナルなクリーンアップ。
- **推奨対応**: コンテキストキャンセルでリスナーを停止する処理を追加する。

#### L-3: Executor 遅延初期化のバックオフ不足

- **ファイル**: `dispatcher.go:100-125`
- **問題**: Executor の遅延初期化に指数バックオフがなく、連続失敗時にリソースを消費する。
- **Codex 見解**: 現状問題なし。
- **推奨対応**: 現状維持。問題が顕在化した場合にバックオフを追加。

#### L-4: shutdown 中のデバウンスコールバック

- **ファイル**: `queue_debounce.go:68-73,99`
- **問題**: シャットダウン中にデバウンスコールバックが実行される可能性がある。
- **Codex 見解**: テスト専用の問題。
- **推奨対応**: テスト環境でのみ影響するため、テストの安定性改善として対応。

#### L-5: Quality Gate の TOCTOU

- **ファイル**: `dispatcher.go:526`
- **問題**: Quality Gate のチェックと適用の間に TOCTOU が存在する。
- **Codex 見解**: 三相設計と整合しており、ドキュメント化を推奨。
- **推奨対応**: 設計上の意図をドキュメントに記載する。

#### L-6: Stdin Temp File の未クリーンアップ

- **ファイル**: `cmd_plan.go:75`
- **問題**: stdin から読み込んだ一時ファイルが適切にクリーンアップされない場合がある。
- **Codex 見解**: 起動時の掃除処理を推奨。
- **推奨対応**: Daemon 起動時に既存の一時ファイルを削除する処理を追加する。

#### L-7: GC 中の全体 mutex 保持

- **ファイル**: `manager.go:902-1025`
- **問題**: GC 処理中に全体の mutex を保持し続けるため、他の操作がブロックされる可能性がある。
- **Codex 見解**: 実測で競合がなければ変更不要。
- **推奨対応**: 現状維持。パフォーマンス問題が顕在化した場合に対応。

#### L-8: CircuitBreaker ProgressTimeoutMinutes のデフォルト値欠落

- **ファイル**: `config.go:260-264`
- **問題**: `CircuitBreaker.ProgressTimeoutMinutes` にコード上のデフォルト値が設定されておらず、テンプレートのみにデフォルトが存在する。テンプレートを使用しない場合にゼロ値となる。
- **Codex 見解**: 実バグ。コード側にもデフォルト値を設定すべき。
- **推奨対応**: `applyDefaults()` 関数内でデフォルト値を設定する処理を追加する。

#### L-9: Persona ファイル読み込み失敗時のサイレントフォールバック

- **ファイル**: `persona/persona.go:73-74`
- **問題**: ペルソナ定義ファイルの読み込みに失敗した場合、エラーログなしにフォールバックする。デバッグ時に原因特定が困難になる。
- **Codex 見解**: 可観測性バグ。ログ出力が欠落している。
- **推奨対応**: 読み込み失敗時に警告レベルのログを出力する。

## 5. 推奨アクション（優先順位付き）

### P0: 即時対応（Critical/High）

1. **C-1 修正**: `manager.go` の `PublishToBase` に dirty check を追加し、未コミット変更がある場合はエラーを返す
2. **H-1 修正**: branch 削除失敗時のログ出力とリコンシリエーション機構を追加
3. **H-2 修正**: GC の `CreatedAt` パース失敗時に `mtime` フォールバックとログ出力を追加

### P1: 計画的対応（Medium）

4. **M-2 修正**: Heartbeat Lease の TOCTOU を解消するアトミック操作への変更
5. **M-1 改善**: 結果処理ループを時間ベースに変更
6. **M-3 修正**: Priority 値のクランプ処理を追加

### P2: 改善推奨（Low）

7. **L-8 修正**: `ProgressTimeoutMinutes` のデフォルト値をコードに追加
8. **L-9 修正**: ペルソナ読み込み失敗時のログ出力追加
9. **L-6 改善**: 一時ファイルの起動時クリーンアップ
10. その他 Low 項目は問題が顕在化した場合に対応

### P3: 任意（Info）

11. 設計意図のドキュメント化（I-1, I-3, I-4）
